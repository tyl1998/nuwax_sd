# 接口层与后端交互

## 1. 统一请求入口

前端请求底座主要看 [src/services/common.ts](../../nuwax/src/services/common.ts)。

这里最重要的几个点：

### 所有请求都会拼接 `BASE_URL`

请求拦截器里直接做了：

- `process.env.BASE_URL + url`

这说明：

- 页面代码里大多写的是相对 API 路径
- 真正连哪个后端，由运行环境变量决定

### 自动注入认证头

会从本地读取：

- `ACCESS_TOKEN`

并注入：

- `Authorization: Bearer <token>`

### 自动注入语言头

会带上：

- `Accept-Language`

所以后端接口层本身已经和国际化联动。

## 2. 错误处理是统一兜底的

[src/services/common.ts](../../nuwax/src/services/common.ts) 还统一处理了：

- 未登录
- 重定向登录
- 智能体不存在
- 沙箱测试异常
- 网络错误
- 请求超时

这意味着：

- 很多页面并不会自己重复写登录失效逻辑
- 错误提示风格在底层就已经统一

## 4. 前端和后端的交互主轴

如果从页面和 service 对应起来看，主交互链路大概是：

### 用户与权限

- 登录信息
- 用户信息
- 菜单权限
- 租户配置

### 对话与智能体

- 智能体详情
- 创建会话
- 临时会话
- 历史会话

### 工作空间资源

- 插件
- 工作流
- MCP
- 知识库
- 数据表
- 页面开发

### 广场与开放能力

- 发布内容
- 订阅
- 订单
- API Key
- 用量与收益

### 系统管理

- 日志
- 菜单权限
- 任务
- 模型监控

## 5. 启动时就会向后端拉什么

重点看 [src/app.tsx](../../nuwax/src/app.tsx)。

启动阶段最关键的交互包括：

- 获取用户信息
- 同步语言
- 查询菜单

而 [src/layouts/index.tsx](../../nuwax/src/layouts/index.tsx) 启动时还会拉：

- 广场分类
- 工作空间列表
- 菜单数据

这说明页面一旦进入主应用，后端就不只是提供“页面点击后的接口”，而是在初始装配阶段就深度参与。

## 6. 鉴权与租户控制如何影响后端交互

重点看 [src/wrappers/authWithLoading.tsx](../../nuwax/src/wrappers/authWithLoading.tsx)。

它会：

- 校验登录态
- 拉租户配置
- 根据租户配置判断某些路由是否可见

所以前后端之间不只是“拿数据渲染页面”，还有：

- 登录态协同
- 租户开关协同
- 权限菜单协同

## 7. 一个很重要的额外信号：AI Agent / AppDev

[AGENTS.md](../../nuwax/AGENTS.md) 专门描述了：

- SSE
- AI Agent 消息流
- AppDev Web IDE
- 文件树
- 开发服务器
- 预览系统

这说明前后端交互里还包含一类特殊链路：

- 普通 REST API
- 流式 AI 消息通道
- 开发环境保活/状态检查

这也是为什么 [src/services/common.ts](../../nuwax/src/services/common.ts) 会特意把下面这些 API 设成静默请求：

- 开发页面保活
- 开发页面启动
- 开发页面重启
- 远程桌面容器保活
- 沙箱状态检查

## 8. 从接口层角度如何理解这个项目

一句话总结：

`nuwax` 前端是一个强后端驱动的业务平台前端，菜单、权限、租户、资源、对话、AI 流式能力都和后端深度绑定；它不是一个“前端本地状态很重、后端很薄”的 SPA。
