# 部署与打包

## 1. 这是一个标准前端静态产物部署模型

从 [Dockerfile](../../nuwax/Dockerfile) 可以直接看出来：

1. 用 Node 镜像安装依赖并构建
2. 产出 `dist`
3. 再切到 Nginx 镜像托管静态资源

所以这个前端最终部署形态不是 Node SSR，而是：

- 前端静态构建产物
- Nginx 托管

## 2. 构建阶段做了什么

重点看 [Dockerfile](../../nuwax/Dockerfile)：

- 安装依赖
- `yarn build:prod`
- 拷贝 `dist` 到 nginx
- 使用 [k8s/default.conf.template](../../nuwax/k8s/default.conf.template) 生成最终 nginx 配置

这里说明两件事：

1. 当前官方镜像流程偏传统前端静态部署
2. 运行时域名通过模板替换，而不是写死在源码里

## 3. Nginx 层主要承担什么

[k8s/default.conf.template](../../nuwax/k8s/default.conf.template) 最重要的是：

- `try_files ... /index.html`

这说明它已经按 SPA 路由刷新场景配置好了。

所以像：

- `/space/:spaceId/...`
- `/system/...`
- `/square/...`

这种前端路由，刷新时仍会正确回到 `index.html`。

## 4. Kubernetes 部署样例说明了什么

仓库自带了：

- [k8s/config/frontend-deployment.yaml](../../nuwax/k8s/config/frontend-deployment.yaml)
- [k8s/config/frontend-service.yaml](../../nuwax/k8s/config/frontend-service.yaml)
- [k8s/config/ingress.yaml](../../nuwax/k8s/config/ingress.yaml)

这说明官方已经默认把它当作：

- 一个独立前端 Deployment
- 暴露成 ClusterIP Service
- 再由 Ingress 对外暴露

## 5. 这个前端和后端是怎么对接的

重点不在 Dockerfile，而在：

- [src/services/common.ts](../../nuwax/src/services/common.ts)

因为接口最终会走：

- `process.env.BASE_URL + url`

所以部署时真正要确认的关键参数，不是“前端打不打得出来”，而是：

- `BASE_URL` 指向哪个后端
- 域名与路由是否对应
- `/m/` 与 PC 站点是否共域或有对应跳转策略

## 6. 部署角度的几个重要结论

### 它是纯前端静态站点

这意味着：

- 平台接入成本相对低
- 不需要 Node 常驻渲染服务

### 它不是一个简单官网静态站

因为它依赖：

- 登录态
- 菜单权限
- 租户配置
- API 基础地址
- PC/H5 双向跳转规则

所以部署时仍然要把运行环境参数和域名策略配对好。

### 它可以比较容易接入公司内部平台

如果公司平台支持：

- Docker 镜像托管
- Nginx 静态服务
- Ingress / 域名配置
- 环境变量注入

那么这个前端本身比 `nuwax-cli`、后端中间件体系更容易迁移。

## 7. 一个需要额外注意的点

[deploy_test.sh](../../nuwax/deploy_test.sh) 说明这个仓库曾经存在“构建产物直接提交到分支”的发布习惯。

这类流程和标准 CI/CD 平台部署并不完全一样。

所以后续如果接入公司内部平台，建议优先采用：

- CI 构建镜像
- 平台部署镜像

而不是继续沿用“本地构建 dist 再推分支”的方式。

## 8. 从部署角度怎么理解这个项目

一句话总结：

`nuwax` 前端本身是容易平台化的，因为它最终只是静态站点；真正要重点确认的是域名、`BASE_URL`、PC/H5 跳转规则，以及和后端登录/权限体系的联动。
