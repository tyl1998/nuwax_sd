# 模型连接「免拼接 URL」改造方案

## 1. 问题背景

目前前端、后端在连接第三方模型时，**默认采用「基址 + 请求后缀」拼接的方式**生成最终转发地址。对于标准 OpenAI 兼容网关（`https://api.openai.com/v1` + `/chat/completions`）没问题，但部分第三方厂商提供的地址是**固定的自定义全路径**（例如 `https://tparty.com/abc/xyz` 或带私有前缀的完整 endpoint），无法用「基址路径 + 进来的请求后缀」拼出来，导致 404 / 路由错误。

诉求：在**配置模型**时增加一个开关（「无需拼接，使用设置 URL」）。后端转发识别到该开关后，**直接使用配置中的完整 URL，不再做任何路径拼接**。

## 2. 现状：当前拼接发生在哪

整条模型调用链路如下（以 OpenAI 协议为例）：

```
前端 → /api/proxy/model/{modelId}/v1/chat/completions → 模型代理(Netty) → 第三方后端 baseUrl
```

### 2.1 基址生成（后端）

`platform-system/system-web/.../TenantController.java:108`

```java
tenantConfigDto.setBaseModelApiUrl(siteUrl + "/api/proxy/model");
```

生成 `baseModelApiUrl`，前端用它拼出模型调用入口。**这一段是代理路由前缀，不能去掉**（否则代理不知道转发到哪个模型配置）。

### 2.2 前端拼模型地址（仅展示/复制用）

`nuwax/src/pages/MorePage/ModelPermissions/index.tsx:76`

```ts
return `${baseModelApiUrl}/${modelId}`;
```

`resolveCopyableInterfaceUrl`（同文件 83 行）对 OpenAI 协议再补 `/v1`。**这是给用户复制的外部 OpenAPI 地址，不是实际转发拼接地**。

### 2.3 真正的转发拼接（后端 · 核心问题点）

请求进入模型代理 `HttpProxyRequestHandler.channelRead`，在 `app-platform-model-proxy/.../proxy/HttpProxyRequestHandler.java:140-157`：

```java
URL url = new URL(backendModelDto.getBaseUrl());
...
request.setUri(url.getPath() + replaceModelPrefix(request.uri(), url.getPath()));
request.headers().set("Host", url.getHost());
```

- `backendModelDto.getBaseUrl()` 来自 `ModelConfigDto.ApiInfo.url`（第三方后端地址）。
- `replaceModelPrefix`（511 行内 `replaceModelPrefix`/`hasVersion`）把进来的 `/api/proxy/model/{id}/v1/xxx` 剥掉前缀，得到 `/xxx`，再拼到 `baseUrl` 的 path 后。

最终：`https://api.openai.com/v1` + `/chat/completions` → `https://api.openai.com/v1/chat/completions`。✅

但当 `apiInfo.url` 是固定全路径（如 `https://tparty.com/abc/xyz`，请求进来想走 `/abc/xyz` 而不是 `/abc/xyz` + 后缀）时，拼接结果错误。❌

### 2.4 配置数据来源

- 模型配置实体：`app-platform-agent/.../adapter/dto/config/ModelConfigDto.java`，内部类 `ApiInfo { url, key, weight }`。
- `HttpProxyRequestHandler` 用的 `baseUrl` 由 `ModelApiProxyConfigServiceImpl.getBackendModelConfig`（同一模块 application 层）从 `ModelConfigDto.ApiInfo.getUrl()` 取出，经 `UserAccessKeyConfig.setModelBaseUrl(...)` → 落到 `BackendModelDto.baseUrl`。

## 3. 改造目标

- 在模型配置增加布尔开关 **`useFullUrl`（无需拼接，使用设置 URL）**。
- 开启时：`HttpProxyRequestHandler` 转发到第三方时，**直接用 `apiInfo.url` 的完整 path，不再追加请求后缀**，也不做 `replaceModelPrefix` 处理。
- 关闭时：保持现有拼接行为，完全向后兼容。

> 说明：前端第 2.2 节的 `/api/proxy/model/{id}` 是代理路由前缀（用于定位模型配置），**不能去掉**；「免拼接」只针对第 2.3 节后端对第三方地址的转发拼接。

## 4. 方案设计

### 4.1 配置开关存储位置

`ModelConfigDto.ApiInfo`（`ModelConfigDto.java`）内增加字段：

```java
@Schema(description = "是否使用完整URL(免拼接)：true=直接使用url作为完整目标地址，不再追加请求路径")
private Boolean useFullUrl;
```

`ApiInfo` 最终序列化为 `model_config` 表的 `api_info_list` JSON 列，无需新增表/列，向后兼容（缺省 null 视为 false）。

### 4.2 开关透传链路

`UserAccessKeyConfig`（agent 模块 AK 配置）增加同名字段 `useFullUrl`，在以下位置透传：

1. `ModelApiProxyConfigServiceImpl.getBackendModelConfig`：`setModelBaseUrl` 同级增加 `setUseFullUrl(modelInfoDto...getUseFullUrl())`。
2. `BackendModelDto`（`model-proxy-sdk/.../dto/BackendModelDto.java`）增加 `Boolean useFullUrl` 字段，并在 `getBackendModelConfig` 组装 `BackendModelDto` 时赋值。
3. 代理读取 `backendModelDto.getUseFullUrl()` 判定。

### 4.3 代理转发逻辑改造（核心）

`HttpProxyRequestHandler.java:140-157`，将固定拼接改为条件分支：

```java
URL url = new URL(backendModelDto.getBaseUrl());
String targetPath;
if (Boolean.TRUE.equals(backendModelDto.getUseFullUrl())) {
    // 免拼接：直接使用配置 URL 的完整 path，忽略进来的请求后缀
    targetPath = url.getPath();
    if (!targetPath.startsWith("/")) {
        targetPath = "/" + targetPath;
    }
} else {
    // 原逻辑：基址路径 + 剥离代理前缀后的请求后缀
    targetPath = url.getPath() + replaceModelPrefix(request.uri(), url.getPath());
}
// host / 端口 / scheme 解析保持不变
...
request.setUri(targetPath);
request.headers().set("Host", url.getHost());
```

> `host` / `port` / `scheme` / `Authorization` 替换逻辑保持不变，仅 `request.setUri(...)` 的取值受开关影响。

### 4.4 前端（可选，本期建议不做）

前端 `ModelPermissions/index.tsx` 复制外部地址时仍走 `/api/proxy/model/{id}`，保持代理路由一致，**本期不改动**。仅在「配置模型」表单（`nuwax/src/pages/SpaceLibrary/CreateModel` 等）增加 `useFullUrl` 开关的 UI 控件，把值写进 `apiInfoList[].useFullUrl` 即可（前后端字段同名透传）。

## 6. 边界与风险

1. **默认关闭 = 旧行为**：`useFullUrl` 为 null/false 时逻辑完全不变，存量模型无感。
2. **免拼接时 body 的 `model` 字段校验**：`HttpProxyRequestHandler` 在 `LastHttpContent` 处校验 `jsonObject.model == currentBackendModel`（`CustomPageChatFluxServiceImpl` 同款逻辑附近，约 295 行），免拼接不改该校验，第三方仍按配置 `model` 字段匹配，不受影响。
3. **URL 必须含 path**：`useFullUrl=true` 时若 `apiInfo.url` 只填了域名（无 path），`url.getPath()` 为空，需兜底为 `/`，避免转发到空路径。文档需提示用户填写完整 endpoint。
4. **AK 缓存**：`BackendModelDto` 缓存 60s（`redisUtil.set(..., 60)`），开关变更最多 1 分钟生效，可接受。
5. **DB 兼容性**：`api_info_list` 为 JSON 列，新增字段老数据缺省为 null，反序列化安全。

## 7. 验证方式

1. 标准模型（OpenAI，`useFullUrl=false`）：行为不变，`/api/proxy/model/{id}/v1/chat/completions` 正常转发。
2. 自定义全路径第三方（如 `https://tparty.com/abc/xyz`，`useFullUrl=true`）：代理直接转发到 `https://tparty.com/abc/xyz`，不再追加后缀；用 curl 对该地址发请求验证返回正确。
3. 边界：只填域名时兜底 `/`，不出现空 path 转发。
4. 观察 `requestContext` 日志中 `backendUrl` / `backendBaseUrl` 字段与预期一致。
