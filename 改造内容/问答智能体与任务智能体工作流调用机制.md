# 问答智能体与任务智能体工作流调用机制

## 1. 智能体类型概述

系统定义了两种智能体类型（`AgentTypeEnum`）：**问答型**（`ChatBot`）和 **通用型**（`TaskAgent`）。

- **问答型（ChatBot）**：适合知识问答、智能客服等快问快答场景。
- **通用型（TaskAgent）**：适合应用开发、深度调研、数据分析、演示文稿制作等复杂任务场景，拥有独立执行电脑（远程桌面），消耗更多资源。

两种类型的智能体**共用同一套工作流组件配置和调用判定逻辑**，区别在于 UI 布局和附加能力（文件树、远程桌面、技能等）。

---

## 2. 工作流作为智能体组件

工作流是智能体的一个**组件类型**（`AgentComponentTypeEnum.Workflow`），位于"工具"分类中，与插件（Plugin）、知识库（Knowledge）、MCP 等并列。

### 2.1 添加工作流到智能体

在 `AgentArrangeConfig` 组件中，工作流作为一个可折叠面板展示。用户通过"添加工作流"按钮打开 `Created` 弹窗，选择已有的工作流并将其添加到智能体中。

添加后，每条工作流组件包含以下配置（`AgentComponentWorkflowUpdateParams.bindConfig`）：

| 配置项 | 类型 | 说明 |
|--------|------|------|
| `invokeType` | `InvokeTypeEnum` | 调用方式（核心判定依据） |
| `inputArgBindConfigs` | `BindConfigWithSub[]` | 入参绑定配置 |
| `outputArgBindConfigs` | `BindConfigWithSub[]` | 出参绑定配置 |
| `async` | `DefaultSelectedEnum` | 是否异步执行 |
| `asyncReplyContent` | `string` | 异步执行回复内容 |
| `directOutput` | `OutputDirectlyEnum` | 是否直接输出 |
| `defaultSelected` | `DefaultSelectedEnum` | 手动模式下是否默认选中 |
| `cardArgsBindConfigs` | `CardArgsBindConfigInfo[]` | 卡片参数绑定信息 |

### 2.2 问答型 vs 通用型的组件差异

在 `AgentArrangeConfig/index.tsx:1443-1459` 中，`Created` 弹窗的标签页按智能体类型做过滤：

- **问答型**：隐藏"技能"（Skill）和"子智能体"（Agent）标签
- **通用型**：仅隐藏"子智能体"（Agent），保留"技能"（Skill）

此外，通用型额外显示以下配置面板：
- 技能（Skills）配置面板
- 允许 @技能（Allow @Skill）开关
- 允许私有沙箱（Allow Private Sandbox）开关
- 隐藏远程桌面（Hide Remote Desktop）开关
- 开场白（Opening Remarks）配置
- 锚点侧边栏（Anchor Sidebar）导航

## 3. 调用方式（InvokeType）— 核心判定逻辑

工作流的**调用方式**（`InvokeTypeEnum`）是决定何时调用工作流的核心配置，包含四种模式：

| 枚举值 | 中文名 | 行为描述 |
|--------|--------|----------|
| `AUTO` | 自动调用 | 用户每次发送消息后都会触发调用一次 |
| `ON_DEMAND` | 按需调用 | 由模型根据任务情况决定是否需要调用 |
| `MANUAL` | 手动选择 | 由用户决定是否使用该工具 |
| `MANUAL_ON_DEMAND` | 手动选择+按需调用 | 用户选择后，由模型根据任务情况决定是否调用；用户不选择则不会调用 |

### 3.1 AUTO（自动调用）

- 工作流**无条件跟随每次用户消息触发**。
- 适合需要固定预处理/后处理的场景（如每次对话先检索再回答）。

### 3.2 ON_DEMAND（按需调用）

- 工作流不会自动触发，而是由**后端 LLM 模型根据对话上下文**判断是否需要调用。
- 模型基于工具描述、函数定义、当前对话内容等因素自行决定调用时机。
- 适合有条件需要的工具（如天气查询、计算器）。

### 3.3 MANUAL（手动选择）

- 工作流不会自动触发，**用户需主动勾选**该工具后才在本次对话中启用。
- 前端通过 `useSelectedComponent` hook 管理用户已选中的组件列表。
- `defaultSelected` 配置决定该工具是否**默认被勾选**。
- 适合用户按需决定是否使用的工具。

### 3.4 MANUAL_ON_DEMAND（手动选择+按需调用）

- 用户先手动选择是否启用该工具，选择后**再由模型决定是否实际调用**。
- 相比 `MANUAL`，用户选中后模型仍有决策权决定不调用。
- 相比 `ON_DEMAND`，模型仅在被选中的工具范围内决策。

## 4. 调用判定流程（前端视角）

### 4.1 配置阶段 — 存储在智能体配置中

```
用户编辑智能体
  → AgentArrangeConfig 展示已添加的组件列表
  → 对每个工作流组件设置 invokeType 等参数
  → 通过 apiAgentComponentWorkflowUpdate 保存到后端
```

### 4.2 运行时 — 前端将选择传递给后端

当用户在对话框发送消息时，`onMessageSend` 构造 `ConversationChatParams` 发送到后端：

关键字段：
- `selectedComponents` — 用户手动选中的组件列表（仅对 MANUAL / MANUAL_ON_DEMAND 模式有效）
- `skillIds` — 技能 ID 列表（仅通用型智能体）
- `message` — 用户消息内容

发送方式：通过 SSE（Server-Sent Events）连接至 `POST /api/agent/conversation/chat`。

### 4.3 运行时 — 后端处理与流式响应

**判定工作流是否调用完全由后端/LLM 完成**，前端仅负责展示结果。

后端通过 SSE 事件流推送执行状态，前端在 `handleChangeMessageList` 中处理：

| SSE 事件类型 | 说明 |
|-------------|------|
| `PROCESSING` | 执行过程状态更新（含 toolCall 执行进度） |
| `MESSAGE` | LLM 流式输出文本（含思考过程/回答/工作流过程输出） |
| `FINAL_RESULT` | 最终结果（通用型触发文件树刷新） |
| `ERROR` | 异常信息 |

`PROCESSING` 事件中的关键字段：

```typescript
interface ProcessingInfo {
  type: AgentComponentTypeEnum;  // Workflow / Plugin / MCP / ToolCall 等
  status: ProcessingEnum;        // EXECUTING / FINISHED / FAILED
  name: string;                  // 组件名称
  executeId: string;             // 执行 ID
  result: ExecuteResultInfo;     // 执行结果
  cardBindConfig?: CardBindConfigInfo;  // 卡片绑定配置
  cardData?: CardDataInfo[];     // 卡片数据
}
```

### 4.4 通用型智能体特有的后处理

`FINAL_RESULT` 事件后，通用型智能体会触发额外的前端操作：

```typescript
// conversationInfo.ts:989-1013
if (params.conversationId &&
    conversationInfoRef.current?.agent?.type === AgentTypeEnum.TaskAgent) {
  // 1. 刷新文件树 — 确保最新生成的文件可见
  await handleRefreshFileList(params.conversationId);
  // 2. 提取任务结果，若有文件则打开预览视图
  const taskResult = extractTaskResult(data.outputText);
  if (taskResult.hasTaskResult && taskResult.file) {
    openPreviewView(params.conversationId);
  }
}
```

此外，通用型智能体还在运行时处理以下事件：
- **OPEN_DESKTOP** 事件：打开远程桌面视图
- **ToolCall** 事件：刷新文件树（当文件树已打开时）

## 5. 完整调用流程图

```
┌─────────────────────────────────────────────────────┐
│                   智能体编辑阶段                        │
│  问答型 / 通用型                                      │
│    → 添加工作流组件（apiAgentComponentWorkflowUpdate） │
│    → 配置 invokeType / 参数绑定 / 卡片绑定等          │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│                   用户发送消息                         │
│  onMessageSend()                                     │
│    → 构造 ConversationChatParams                     │
│      ├ selectedComponents（手动选中的组件）           │
│      ├ skillIds（仅通用型）                           │
│      └ message（用户输入）                            │
│    → POST /api/agent/conversation/chat（SSE连接）     │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│               后端/LLM 决策层                          │
│                                                      │
│  根据以下因素判定是否调用工作流：                       │
│  ┌─────────────────────────────────────────────┐     │
│  │ invokeType = AUTO         → 每次必调         │     │
│  │ invokeType = ON_DEMAND    → LLM 自行决定     │     │
│  │ invokeType = MANUAL       → 仅用户选中时触发  │     │
│  │ invokeType = MANUAL_ON_DEMAND → 用户选中后    │     │
│  │                               LLM 决定是否调  │     │
│  └─────────────────────────────────────────────┘     │
│                                                      │
│  调用时：执行工作流并流式返回 PROCESSING 事件          │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│              前端 SSE 事件处理                        │
│                                                      │
│  PROCESSING 事件：                                    │
│    → 更新 processingList 展示执行进度                 │
│    → EXECUTING: 处理扩展页面/链接打开                 │
│    → FINISHED:  处理卡片数据展示                      │
│                                                      │
│  MESSAGE 事件：                                       │
│    → THINK: 更新思考过程                              │
│    → QUESTION/通用: 流式展示回答文本                   │
│                                                      │
│  FINAL_RESULT 事件：                                  │
│    → 通用型: 刷新文件树 + 打开预览视图                 │
│    → 问答型: 无额外操作                               │
└─────────────────────────────────────────────────────┘
```

## 7. 总结

- **问答型与通用型智能体在工作流调用机制上完全一致**，都使用 `InvokeTypeEnum` 判定调用时机。
- **判定工作流是否调用的核心逻辑在后端/LLM 侧**，前端仅负责配置存储（invokeType、参数绑定）、用户选择传递（selectedComponents）和执行结果展示。
- 通用型智能体相比问答型在**工作流调用后**有更多副作用处理：文件树刷新、远程桌面打开、预览视图展示等。
- `invokeType` 的四种模式（AUTO/ON_DEMAND/MANUAL/MANUAL_ON_DEMAND）覆盖了从固定调用到完全人工选择的不同场景。
