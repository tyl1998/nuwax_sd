# 知识库更换向量模型「按维度重建」改造方案

## 1. 问题背景

知识库编辑时更换「向量模型（embeddingModelId）」，后端会触发向量集合的重建与全量重向量化。但原实现存在两个问题：

1. **过度重建**：只要 `embeddingModelId` 的 id 变化，就无条件 `deleteCollection` + 全量重算。即使新旧模型**维度相同**（例如同为 1536 维、仅换厂商或仅改 key/url），旧向量在维度上仍可复用，强制重建会白白清空数据、消耗资源，且重建期间该知识库检索不可用。
2. **维度冲突 Bug（致命）**：重建采用「先 `deleteCollection` 删旧集合 → 再异步 `batchAddEmbeddingQa` 重向量化」的方式，而真正按新维度建集合是**懒重建**（写入时 `addEmbeddingQaForBatch` 发现集合不存在才 `initAndCheckCollection`）。删除旧集合到按新维度建好之间存在**时序窗口**：若此时有查询或别处写入，用新模型（如 2048 维）往尚未按新维度建好的集合（或旧维度 1536 集合）写向量，就会抛：

   ```
   io.milvus.exception.ParamException: Incorrect dimension for field 'vector':
   dimension: 2048 is not equal to field's dimension: 1536
   ```

## 2. 现状：原重建逻辑在哪

编辑知识库 → `KnowledgeConfigController.update` → `KnowledgeConfigApplicationService.updateInfo`
（`app-platform-knowledge/.../core-application/.../KnowledgeConfigApplicationService.java`）。

原逻辑（已删除的 debug 版）：

```java
if (existObj.getEmbeddingModelId() != model.getEmbeddingModelId()) {
    // 1. 标记问答 has_embedding = 0
    // 2. deleteCollection(kbId)                // 删旧集合
    // 3. 异步 batchAddEmbeddingQa(modelList)   // 用新模型重向量化（集合在此才懒重建）
}
return knowledgeConfigDomainService.updateInfo(model, userContext);  // 更新 DB
```

配套写入路径：`MilvusDomainService.addEmbeddingQaForBatch`（第 231-275 行）在插入前若集合不存在，会用 `kbConfig.getEmbeddingModelId()`（已是新模型）懒重建集合。**正是这个“删除后、异步懒重建”的窗口导致了维度冲突。**

## 3. 为什么更换模型必须重建（设计动机）

知识库的**写入（向量化）**与**查询（检索）**使用的是**同一个 `embeddingModelId`**：

- 写入：`KnowledgeQaSegmentDomainService` 用 `knowledgeConfig.getEmbeddingModelId()` 把问答向量化进 Milvus。
- 查询：`KnowledgeQaSegmentDomainService.search`（第 169-193 行）→ `QAQueryDto.convertTEmbeddingDto`（第 34-37 行）用 `knowledgeConfig.getEmbeddingModelId()` 把**用户问题**编码成查询向量，再去 Milvus 比对。

若存储向量与查询向量来自不同模型/不同向量空间，相似度计算会失真甚至维度对不上。因此：**当向量空间（特别是维度）真正变化时，旧向量必须重算**，这是重建的合理动机。

## 4. 改造目标

1. **按维度决定是否重建**：仅当新旧向量模型**维度不同**时才重建集合并重向量化；维度相同则复用旧向量，跳过重建。
2. **原子重建**：删除旧集合后，**同步**按新模型维度建好空集合，再异步重向量化，消除“新维度向量写入旧维度集合”的竞态窗口。
3. **保守兜底**：任一模型维度无法获取时，按“维度变化”处理执行重建，避免潜在维度冲突。
4. **清理调试代码**：移除原 `System.out.println` 调试日志。

## 5. 方案设计

### 5.1 维度对比（核心判断）

在 `updateInfo` 的模型变更分支中，通过 `modelApplicationService.queryModelConfigById(id).getDimension()` 取得新旧模型维度：

```java
Integer oldDim = queryModelDimension(existObj.getEmbeddingModelId());
Integer newDim = queryModelDimension(model.getEmbeddingModelId());

// 仅当新旧维度都已知且相等时才跳过重建；否则（维度不同或未知）一律重建
boolean sameDimension = oldDim != null && newDim != null && oldDim.equals(newDim);
if (sameDimension) {
    // 维度未变：直接更新配置，复用旧向量
    return this.knowledgeConfigDomainService.updateInfo(model, userContext);
}
```

### 5.2 原子重建（消除竞态）

仅当维度变化（或未知）时执行，且**先同步建集合再异步重算**：

```java
// 标记问答待向量化
jdbcTemplate.update("update knowledge_qa_segment set has_embedding = 0, created = now() where kb_id = ?", ...);

// 删旧集合 + 同步按新模型维度重建空集合（关键：在异步重向量化之前完成）
vectorDBService.deleteCollection(model.getId());
vectorDBService.initAndCheckCollection(model.getId(), model.getEmbeddingModelId());

// 异步用新模型重向量化
List<Long> ids = jdbcTemplate.queryForList("select id from knowledge_qa_segment where kb_id = ?", ...);
List<KnowledgeQaSegmentModel> modelList = knowledgeQaSegmentRepository.queryListByIds(ids);
threadTenantUtil.obtainOtherScheduledExecutor().execute(
    () -> knowledgeQaSegmentDomainService.batchAddEmbeddingQa(modelList, userContext));
```

> 由于 `initAndCheckCollection` 在删除后**同步**以新模型维度建好空集合，`batchAddEmbeddingQa` 写入时集合已是正确的新维度，不再出现 `2048 != 1536` 的冲突。

### 5.3 维度查询辅助方法

```java
private Integer queryModelDimension(Long modelId) {
    if (modelId == null) return null;
    try {
        var modelConfig = modelApplicationService.queryModelConfigById(modelId);
        return modelConfig != null ? modelConfig.getDimension() : null;
    } catch (Exception e) {
        log.warn("Query embedding model dimension failed, modelId={}", modelId, e);
        return null;
    }
}
```

## 7. 行为对照

| 场景 | 改造前 | 改造后 |
|------|--------|--------|
| 1536 维 → 2048 维（如 doubao-embedding-vision） | 重建（但可能维度冲突报错） | 同步重建，无冲突 |
| 同维度换模型（1536 → 1536 不同厂商） | 无条件重建（浪费） | 跳过重建，复用旧向量 |
| 仅改 key/url，模型本身未变 | 重建（无意义） | 跳过重建 |
| 维度查询失败（未知） | 重建 | 保守重建（避免冲突） |
| 模型 id 未变化 | 不重建 | 不重建 |

## 8. 边界与风险

1. **同维度换模型跳过重建的语义前提**：默认假设“同维度向量可混用/复用”。若业务上要求不同厂商同维度向量的向量空间也必须严格隔离，则需改为“维度相同也重建”。当前按“避免无谓清空 + 维度兼容”的更优体验处理。
2. **重建期间短暂检索为空**：维度变化时删除旧集合、重向量化完成前，该知识库语义检索结果会暂时为空（全文检索不受影响）。此为预期行为。
3. **异步重向量化的租户上下文**：沿用原 `TenantRunnable` + `obtainOtherScheduledExecutor`，租户隔离不变。
4. **向后兼容**：仅调整重建触发条件与顺序，未改表结构、接口契约，存量数据无感。

## 9. 验证方式

1. **维度变化（1536→2048）**：编辑知识库换成 `doubao-embedding-vision`，保存后应平滑重建，不再出现 `dimension 2048 != 1536` 报错；重建完成后语义检索正常。
2. **同维度换模型**：编辑知识库换成另一个同维度模型，保存后**不重建**，旧问答 `has_embedding` 保持，检索立即可用，日志打印 `skip rebuild`。
3. **仅改 key/url**：模型 id 不变，完全不触发重建分支。
4. 观察日志关键字：`embedding model changed but dimension unchanged ... skip rebuild` 与 `embedding model changed with dimension changed/null ... rebuild collection`。
