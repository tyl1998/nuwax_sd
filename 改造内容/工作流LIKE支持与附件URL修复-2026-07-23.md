# 工作流 LIKE 查询支持 & TaskAgent 附件 URL 修复 - 2026-07-23

## 一、工作流数据表节点支持 LIKE 查询

### 问题背景

工作流"数据查询"节点的条件选择器不支持 LIKE 模糊匹配。底层 SQL 引擎（Doris/MySQL）完全支持 LIKE，问题出在应用层条件枚举和 SQL 生成器缺少 LIKE 分支。

### 根因分析

| 层面 | 问题 |
|------|------|
| `CompareTypeEnum` 枚举 | 只有 =、!=、>、>=、<、<=、IN、NOT IN、IS NULL、IS NOT NULL，没有 LIKE |
| `TableExecutor.convertCompareType()` | switch 分支只处理上述枚举，无 LIKE 对应的 SQL 操作符 |
| 前端 `tableOptions` 选项 | 条件下拉框没有 LIKE 选项 |
| 底层 SQL 引擎 | 无限制，`SqlParserUtil` 只禁 DDL，LIKE 完全可用 |

已验证：compose RPC 层和"SQL自定义"节点可以直接执行 `LIKE` SQL（测试用例 `ComposeDbTableRpcServiceImplTest.testSelectLikeCustomTable28` 已覆盖）。

### 改动清单

#### 后端（nuwax-backend）

| 文件 | 改动 |
|------|------|
| `app-platform-agent-core-adapter/.../TableNodeConfigDto.java:78` | `CompareTypeEnum` 新增 `LIKE`、`NOT_LIKE` |
| `app-platform-agent-core-infra/.../TableExecutor.java:200` | `convertCompareType()` 新增 `LIKE → " LIKE "`、`NOT_LIKE → " NOT LIKE "` |

#### 前端（nuwax）

| 文件 | 改动 |
|------|------|
| `src/types/enums/common.ts:207` | `CompareTypeEnum` 新增 `LIKE`、`NOT_LIKE` |
| `src/pages/Antv-X6/params.tsx` | `options` 和 `tableOptions` 两个数组都新增 LIKE/NOT_LIKE 选项 |
| `src/pages/Antv-X6/v3/constants/node.constants.ts:91` | `compareTypeMap` 新增 `LIKE → ~`、`NOT_LIKE → !~` |
| `src/constants/node.constants.tsx:93` | `compareTypeMap` 新增 `LIKE → ~`、`NOT_LIKE → !~` |

#### 国际化（i18n）

| 语言文件 | `operatorLike` | `operatorNotLike` |
|---------|----------------|-------------------|
| `zh-CN.ts` | 模糊匹配 | 模糊不匹配 |

| `zh-TW.ts` | 模糊匹配 | 模糊不匹配 |

### 使用方式

工作流"数据查询"节点条件选择器选择 `~`（LIKE）或 `!~`（NOT LIKE），值输入 `%keyword%` 模式：

```
字段: name | 比较: ~ | 值: %test%
→ WHERE name LIKE '%test%'

字段: email | 比较: !~ | 值: %@spam.com
→ WHERE email NOT LIKE '%@spam.com'
```

### 不受影响的场景

| 场景 | 影响 |
|------|------|
| 工作流"SQL自定义"节点 | 无影响，直接执行原始 SQL |
| 智能体对话中模型调用数据表 | 无影响，模型直接生成 SQL |
| 工作流校验逻辑 | 无影响，`checkTableConditionArgs` 只检查 `compareType != null` |
| 已有工作流配置 | 无影响，新枚举值向后兼容 |

### 底层数据库说明

数据表底层是 Apache Doris，通过 MySQL 协议（JDBC）访问。STG 环境无独立 Doris 实例，回退到 MySQL（`172.17.52.69:13306`）。两者都支持 LIKE 语法。

## 二、TaskAgent 聊天附件 URL 修复

### 问题背景

在智能体聊天内上传文件后，传给 TaskAgent 沙箱的附件 URL 仍使用办公网 IP（`10.94.77.42:18080`），沙箱容器在集群内无法访问该地址。

### 根因分析

附件处理链路：

```
用户上传文件
  → FileManagementServiceImpl.uploadFile()  // fileUrl = siteUrl + "/api/f/" + fileKey
  → ConversationApplicationServiceImpl      // iFileAccessService.getFileUrlWithAk(fileUrl, true)
  → S3FileStorageStrategy.generatePresignedUrl()  // 生成 public presigned URL
  → AgentExecutor.buildUserMessage()        // JSON 序列化 attachments 进 <attachment> 标签
  → SandboxAgentClient                      // 发送给沙箱
```

问题在第 3 步：`getFileUrlWithAk(..., true)` 对 S3/MinIO 文件会调用 `generatePresignedUrl()`，该方法将 `S3_ENDPOINT`（集群内 MinIO 地址）替换为 `S3_PUBLIC_ENDPOINT`（办公网地址），导致沙箱收到的是 `10.94.77.42:18080` 的预签名 URL。

### 之前的 MinIO 改造基础

之前已做过以下改造（见 `文件URL改造-2026-07-02.md`）：

- `S3FileStorageStrategy` 支持 `s3.public-endpoint`，外部预签名 URL 会把 internal `S3_ENDPOINT` 替换为 `S3_PUBLIC_ENDPOINT`
- `generateInternalPresignedUrl()` 保留 `S3_ENDPOINT`，不替换为 public endpoint
- `FileUrlResolver.toAbsoluteUrl()` 对云存储 URL 委托 `resolveInternalFileUrl()` 生成内部可达的签名 URL
- `FileManagementServiceImpl.resolveInternalFileUrl()` 解析 `/api/f/s3/...` 或 `/nuwax-files/s3/...` 并重新生成内部签名 URL

### 改动方案

**只改 TaskAgent 执行链路**，不影响前端展示和历史消息。

#### 改动文件

| 文件 | 改动 |
|------|------|
| `ConversationApplicationServiceImpl.java` | 新增 `toInternalAttachmentUrls()` 方法；附件追加 AK 后，TaskAgent 单独改写 `agentContext.attachments` |

#### 改动逻辑

```java
// 附件追加 AK（原有逻辑，不改）
if (tryReqDto.getAttachments() != null) {
    tryReqDto.getAttachments().forEach(attachmentDto ->
        attachmentDto.setFileUrl(iFileAccessService.getFileUrlWithAk(attachmentDto.getFileUrl(), true)));

    // TaskAgent: 把传给沙箱的附件 URL 转为内部可达地址
    if ("TaskAgent".equals(agentConfigDto.getType())) {
        agentContext.setAttachments(toInternalAttachmentUrls(tryReqDto.getAttachments()));
    }
}
```

`toInternalAttachmentUrls()` 复制附件对象，只替换传给沙箱的 `fileUrl`：

```java
private List<AttachmentDto> toInternalAttachmentUrls(List<AttachmentDto> attachments) {
    return attachments.stream().map(attachment -> {
        AttachmentDto internalAttachment = new AttachmentDto();
        internalAttachment.setFileKey(attachment.getFileKey());
        internalAttachment.setFileName(attachment.getFileName());
        internalAttachment.setMimeType(attachment.getMimeType());
        internalAttachment.setFileUrl(FileUrlResolver.toAbsoluteUrl(attachment.getFileUrl()));
        return internalAttachment;
    }).collect(Collectors.toList());
}
```

#### 效果

| 场景 | 之前 | 之后 |
|------|------|------|
| 前端展示的附件 URL | `http://10.94.77.42:18080/api/f/s3/...` | 不变 |
| 会话历史中的附件 URL | `http://10.94.77.42:18080/api/f/s3/...` | 不变 |
| 普通 Agent（ChatBot）附件 URL | `http://10.94.77.42:18080/api/f/s3/...` | 不变 |
| TaskAgent 沙箱内附件 URL | `http://10.94.77.42:18080/api/f/s3/...` | `http://172.17.52.69:9000/nuwax-files/s3/...`（内部签名 URL） |

