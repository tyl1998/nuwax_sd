## 3. 推荐阅读顺序

### 第一步：理解整体架构（1 小时）

1. `README.md` — 快速过一遍，建立架构图
2. `docs/grpc-architecture.md` — gRPC 迁移设计文档，有完整链路图
3. `crates/rcoder/src/main.rs` — 启动流程：Docker 初始化 → 清理任务 → HTTP 服务 → Pingora

### 第二步：跟踪一次请求（2 小时）

4. `handler/computer_chat_handler.rs` — 从 `handle_computer_chat()` 逐行读：
   - 并发保护（pod_creating DashMap）
   - 容器获取/创建
   - IP 二次验证
   - gRPC Chat 调用
5. `service/computer_container_manager.rs` — `get_or_create_container_for_user()`
6. `grpc/chat_client.rs` — `grpc_chat_with_pool()` + HTTP 回退

### 第三步：理解进度流（1.5 小时）

7. `grpc/sse_stream.rs` — `create_grpc_sse_stream()`：
   - GetStatus 预检查
   - SubscribeProgress Server Streaming
   - mpsc channel 解耦
   - `progress_event_to_sse()` 事件转换
8. `grpc/converters.rs` — `from_grpc_progress_event()`（rcoder 侧转换）
9. `agent_runner/grpc/agent_service_impl.rs` — `unified_message_to_progress_event()`（发送侧转换）

### 第四步：了解容器生命周期（1 小时）

10. `cleanup_task/` 目录 — 清理策略、空闲超时、保护期
11. `service/container_sync.rs` — 状态同步任务
12. `handler/pod_handler.rs` — Pod 管理 API（ensure/keepalive/restart）

## 4. 与 nuwax-backend 的对接点

```
nuwax-backend
└── agent/infra/rpc/SandboxAgentClient.java
    ├── chat()         → POST /computer/chat
    ├── progress()     → GET  /computer/progress/{session_id}（SSE）
    ├── cancel()       → POST /computer/agent/session/cancel
    └── keepalive()    → POST /computer/pod/keepalive
```

nuwax-backend 在 `AgentExecutor` 检测到 TaskAgent 时切换到 `SandboxAgentClient`，后者管理 rcoder 的连接。

## 5. 常见问题定位

| 现象 | 检查位置 |
|------|---------|
| 容器创建卡住 | `computer_chat_handler.rs` — pod_creating 等待逻辑（最多 30s）|
| SSE 连接立即关闭 | `sse_stream.rs` — GetStatus 检查，Agent 已 idle → 立即发 SessionPromptEnd |
| gRPC 连接失败 | `channel_pool.rs` — remove() 清理 + 重试逻辑；检查容器 IP 和端口 50051 |
| VNC 无法访问 | `computer_desktop_handler.rs` + Pingora 配置；检查容器内 noVNC :6080 |
| 容器频繁被清理 | `cleanup_task/` — 检查 `idle_timeout_seconds` 和 `container_protection_seconds` |
| 并发请求创建两个容器 | `computer_chat_handler.rs` — pod_creating DashMap 并发保护逻辑 |
