# 智能体测试方案

## 1. 目标

本文档面向 nuwax 技术体系里的智能体能力设计测试方案，重点覆盖 Web 前端、nuwax-backend、mcp-proxy、rcoder、agent_runner、nuwaclaw、nuwax-file-server 等组件组成的智能体链路。

测试目标不是单纯验证模型回答是否正确，而是同时验证：

- 智能体请求链路是否稳定。
- 流式事件是否完整、顺序正确、可被前端正确渲染。
- MCP、插件、知识库、工作流、沙箱等工具调用是否可靠。
- TaskAgent、Computer Agent、ACP 引擎等多类 Agent 能力是否可用。
- 多租户、工作空间、会话、容器和本地引擎之间是否隔离。
- 异常场景下是否能恢复、降级、提示或安全结束。
- 智能体在真实任务中的完成质量是否可度量。

## 2. 测试范围

### 2.1 主服务链路

主服务链路来自部署关系：

```text
nuwax / nuwax-mobile / nuwaclaw
  -> nuwax-backend
  -> mcp-proxy
  -> MySQL / Redis / Milvus
  -> rcoder
  -> agent_runner containers
  -> nuwaxcode / claude-code-acp-ts / run_code_rmcp
```

本阶段优先覆盖：

- `nuwax`：聊天发送、SSE 接收、消息状态、Markdown 渲染、工具进度展示、右侧预览/VNC/文件树联动。
- `nuwax-backend`：AgentExecutor、ModelInvoker、Function Calling 循环、SSE 推流、知识库与记忆注入、TaskAgent 转发。
- `mcp-proxy`：MCP 服务注册、动态路由、协议转换、健康检查、Persistent 自动重启、OneShot 清理。
- `rcoder`：容器创建、Computer Agent 请求、gRPC Chat、SubscribeProgress、SSE 桥接、HTTP 回退。
- `nuwaclaw`：ACP 引擎管理、nuwaxcode/claude-code 切换、warmup、MCP bridge、记忆提取。
- `nuwax-file-server`：项目路径隔离、Computer 工作空间、skills/agents 创建与动态技能保留。

### 2.2 不作为第一阶段重点

- 移动端完整 UI 兼容。
- 大规模模型效果榜单。
- 模型供应商本身的能力横评。
- 完整安全渗透测试。
- 所有插件和所有业务工作流的穷举测试。

这些可以在主链路稳定后作为第二阶段补充。

## 3. 测试分层

### 3.1 协议层测试

验证各组件之间的协议、事件格式和边界行为。

| 模块 | 重点 | 示例 |
| --- | --- | --- |
| backend SSE | 事件类型和顺序 | `MESSAGE`、`PROCESSING`、`FINAL_RESULT`、`HEART_BEAT` 是否完整 |
| rcoder SSE | gRPC 到 SSE 映射 | `thinking`、`chunk`、`tool_use`、`completion`、`error` 是否正确 |
| gRPC | 类型化进度流 | `ProgressEvent.oneof` 是否能覆盖所有事件 |
| MCP | 路径和注册协议 | `/mcp/sse/{id}/sse`、`/mcp/sse/{id}/message`、`/mcp/stream/{id}/mcp` |
| ACP | stdio NDJSON 通信 | 引擎启动、消息发送、响应事件、异常退出 |

期望结果：

- 事件字段稳定。
- 完成事件只出现一次。
- 错误事件可以被消费方识别。
- 流关闭后状态一致。
- 协议异常不会造成前端无限 loading。

### 3.2 链路层测试

验证一次智能体请求从用户输入到最终结果的完整路径。

| 链路 | 入口 | 关键断言 |
| --- | --- | --- |
| ChatAgent 普通对话 | 前端聊天框 / backend chat API | 前端出现用户消息、AI 占位消息、流式文本、完成状态 |
| Function Calling | 模型触发工具调用 | backend 推送 `PROCESSING EXECUTING` 和 `FINISHED`，工具结果进入下一轮模型调用 |
| Knowledge Agent | 绑定知识库对话 | 检索结果注入 prompt，回答能引用相关知识 |
| Memory Agent | 开启长期记忆 | 下轮对话前能检索记忆，对话结束后异步提取 |
| TaskAgent | backend 转发 rcoder | rcoder 返回 session，progress 流最终完成 |
| Computer Agent | `/computer/chat` | 容器按 user 创建，workspace 挂载正确，进度可订阅 |
| Desktop Agent | VNC 视图 | `OPEN_DESKTOP` 事件触发前端切换远程桌面 |

期望结果：

- 不同 Agent 类型都能形成可观测闭环。
- 前端、backend、rcoder 的 session/requestId 能关联。
- 工具调用过程中前端展示不丢、不乱、不重复。
- 任务结束后资源状态可查询。

### 3.3 工具层测试

验证智能体可以可靠调用外部能力。

| 工具类型 | 分发前缀/组件 | 测试点 |
| --- | --- | --- |
| 插件 | `plugin_*` | 参数传递、执行结果、异常返回 |
| 工作流 | `workflow_*` | 流程触发、节点错误、最终结果 |
| MCP | `mcp_*` | 注册、调用、重启、协议转换 |
| 知识库 | `knowledge_*` | 检索策略、topK、阈值、空结果 |
| 数据表 | `compose_*` | 查询参数、权限、结果格式 |
| 沙箱工具 | rcoder / agent_runner | 文件读写、命令执行、取消任务 |
| 文件服务 | nuwax-file-server | 工作空间创建、动态技能推送、路径隔离 |

期望结果：

- 工具成功时，结果能进入模型后续上下文。
- 工具失败时，错误能传递给模型或前端。
- 工具调用不会泄漏其他租户、项目、会话的数据。
- 动态 MCP 或动态技能变更后，下一次请求能生效。

### 3.4 前端体验层测试

验证用户看到的智能体交互是否正常。

重点场景：

- 流式文本逐字输出。
- thinking 块可展开收起。
- Markdown、代码高亮、Mermaid、KaTeX 正常渲染。
- 工具调用进度 custom block 正常展示。
- 生成页面时右侧 PagePreviewIframe 自动打开。
- 打开桌面时右侧 VNC 自动切换。
- 预览模式下工具调用触发文件树节流刷新。
- 用户上滑后不会强制滚动到底部，点击回到底部后恢复。
- 历史消息关闭打字动画，直接展示完整内容。

期望结果：

- 不出现无限 loading。
- 不出现重复 token、丢 token、顺序错乱。
- FINAL_RESULT 后消息状态变为完成。
- 大段 Markdown 或代码块不会卡死页面。

### 3.5 稳定性和异常测试

重点覆盖智能体系统最容易出问题的长连接、容器、进程、工具和模型异常。

| 场景 | 注入方式 | 期望 |
| --- | --- | --- |
| 模型超时 | mock 模型 API 延迟或断开 | backend 结束流并返回可识别错误 |
| SSE 断线 | 前端中断连接后重连 | 已完成任务能补结束状态，未完成任务可继续订阅 |
| MCP 后端退出 | kill MCP 子进程 | Persistent 自动重启，OneShot 清理 |
| gRPC 不可用 | 关闭 agent_runner gRPC | rcoder 尝试 HTTP 回退 |
| 容器 IP 为空 | mock Docker 缓存异常 | rcoder 清理并强制重建 |
| 同 user 并发创建容器 | 并发 `/computer/chat` | 只创建一个容器，第二个等待或复用 |
| 工具执行失败 | 返回错误结果 | 前端展示工具失败，模型可继续解释 |
| 任务取消 | 调用 CancelSession | progress 流收到取消/错误事件，容器不残留活跃任务 |
| 长时间无输出 | 模型或工具沉默 | 心跳维持连接，超时策略生效 |

### 3.6 安全和隔离测试

重点验证智能体执行能力不会越界。

| 对象 | 测试点 |
| --- | --- |
| 多租户项目路径 | `tenantId + spaceId + isolationType + projectId` 组合路径正确 |
| 默认项目路径 | 缺少任一隔离字段时回退到 `PROJECT_SOURCE_DIR/{projectId}` |
| Computer 工作空间 | `COMPUTER_WORKSPACE_DIR/{userId}/{cId}` 隔离 |
| 动态技能 | `.dynamic_add.lock` 标记的技能在 workspace 重建后保留 |
| ACP 引擎环境 | `HOME`、`USERPROFILE`、`CLAUDE_*`、`ANTHROPIC_*` 不读取用户真实环境 |
| MCP 配置 | ACP 下发 MCP 优先于本地配置，变更检测不误判 |
| 日志 | API Key、token、敏感环境变量不进入日志 |
| 文件访问 | Agent 不能读取其他 user/project/session 的工作目录 |

## 4. 用例矩阵

### 4.1 P0 主链路用例

| 编号 | 场景 | 步骤 | 断言 |
| --- | --- | --- | --- |
| AGENT-P0-001 | 普通对话流式输出 | 发送普通问题 | 收到 `MESSAGE`，最终收到 `FINAL_RESULT` |
| AGENT-P0-002 | thinking 输出 | 使用支持 thinking 的模型 | thinking 与 answer 分开展示 |
| AGENT-P0-003 | Function Calling 成功 | 配置一个可用工具并提问 | 出现工具执行中和执行完成事件 |
| AGENT-P0-004 | Function Calling 失败 | 工具返回错误 | 前端展示错误，流正常结束或模型解释失败原因 |
| AGENT-P0-005 | TaskAgent 沙箱任务 | 发送创建文件/运行命令任务 | rcoder 返回 session，progress 最终 completion |
| AGENT-P0-006 | MCP 注册与调用 | 注册 MCP 后触发工具 | 动态路由存在，工具结果返回 backend |
| AGENT-P0-007 | 前端 Markdown 渲染 | 让模型输出代码块、表格、Mermaid | 渲染正确，无明显卡顿 |
| AGENT-P0-008 | 会话完成状态 | 正常对话结束 | AI 消息状态从 Loading/Incomplete 变为 Complete |

### 4.2 P1 能力评测用例

| 编号 | 场景 | 任务 |
| --- | --- | --- |
| AGENT-P1-001 | 代码理解 | 让智能体解释 `nuwax-backend` 的模型调用链路 |
| AGENT-P1-002 | 代码修改 | 让智能体在一个小模块中补充字段或校验 |
| AGENT-P1-003 | 工具选择 | 任务要求必须读取文件、运行命令、调用 MCP |
| AGENT-P1-004 | 多轮修正 | 第一轮给模糊需求，第二轮补充约束，观察是否修正方案 |
| AGENT-P1-005 | 知识库回答 | 上传文档后提问，检查引用是否来自知识库 |
| AGENT-P1-006 | 记忆利用 | 开启长期记忆后，下一轮检查是否使用历史偏好 |
| AGENT-P1-007 | 页面生成 | 让智能体生成一个简单页面，并检查预览事件 |
| AGENT-P1-008 | 失败恢复 | MCP 或命令失败后，观察是否能换路径继续完成任务 |

### 4.3 P2 稳定性用例

| 编号 | 场景 | 目标 |
| --- | --- | --- |
| AGENT-P2-001 | 20 并发普通对话 | 验证 backend SSE 和模型调用池稳定 |
| AGENT-P2-002 | 20 并发 Computer Agent | 验证同 user/不同 user 容器复用和隔离 |
| AGENT-P2-003 | SSE 长连接 30 分钟 | 验证心跳、超时和资源释放 |
| AGENT-P2-004 | MCP 服务反复重启 | 验证 Persistent 自动恢复 |
| AGENT-P2-005 | agent_runner gRPC 间歇失败 | 验证连接池清理和 HTTP 回退 |
| AGENT-P2-006 | 大输出 Markdown | 验证前端增量渲染性能 |
| AGENT-P2-007 | 任务取消后重发 | 验证 session 状态清理 |
| AGENT-P2-008 | 引擎配置频繁变更 | 验证 nuwaclaw 引擎重建和活跃 prompt 保护 |

## 5. 自动化落地建议

### 5.1 第一阶段：链路自动化

优先做接口和协议级自动化，不依赖人工判断模型效果。

建议覆盖：

- backend chat SSE contract test。
- TaskAgent 到 rcoder 的 progress contract test。
- mcp-proxy 动态注册和调用测试。
- rcoder gRPC 事件映射测试。
- nuwax-file-server workspace 隔离测试。

自动化断言重点：

- 事件类型。
- 字段完整性。
- 事件顺序。
- session/requestId 关联。
- 结束条件。
- 错误格式。
- 资源清理状态。

### 5.2 第二阶段：智能体评测集

建立 20 到 30 条固定任务，形成版本间可比较的能力基线。

每条任务记录：

- 输入 prompt。
- 依赖数据或仓库。
- 允许使用的工具。
- 期望产物。
- 自动评分规则。
- 人工复核点。

评分维度：

| 维度 | 说明 |
| --- | --- |
| 完成率 | 是否产出预期结果 |
| 正确性 | 结果是否符合要求 |
| 工具使用 | 是否调用了必要工具，是否误用工具 |
| 稳定性 | 是否中断、超时、无限重试 |
| 最小改动 | 代码任务是否只改必要文件 |
| 可解释性 | 是否能说明依据和验证方式 |
| 成本 | token、耗时、工具调用次数 |

### 5.3 第三阶段：压测和故障注入

在主链路和评测集稳定后加入：

- 并发压测。
- 长连接压测。
- MCP 故障注入。
- gRPC 故障注入。
- 容器创建并发。
- 大文件/大输出。
- 敏感信息日志扫描。

如果只做文档沉淀，保留当前方案文档即可；如果进入自动化阶段，再逐步创建 `cases/fixtures/automation/reports`。

## 7. 第一批建议实施清单

优先顺序建议如下：

1. 建立 P0 主链路用例文档，先覆盖 8 条核心用例。
2. 选定 backend chat SSE 的测试入口，固定一套 mock 或测试模型配置。
3. 选定一个最小 MCP 服务，用于注册、调用、重启测试。
4. 选定一个 TaskAgent 沙箱任务，例如创建文件、读取文件、运行简单命令。
5. 补一组前端手工验收清单，覆盖 thinking、Markdown、工具进度、VNC、文件树刷新。
6. 建立 20 条以内的智能体能力评测 prompt，先做人工评分，稳定后再加自动评分。
7. 将异常注入作为第二轮，不阻塞主链路测试落地。

## 8. 验收标准

第一阶段可以用以下标准判断方案是否落地：

- P0 用例全部可执行，有明确输入、步骤、断言。
- 普通 ChatAgent、Function Calling、TaskAgent、MCP 调用四条主链路至少各有 1 条自动化或半自动化用例。
- 每条 SSE/gRPC/MCP 用例都能断言结束状态，不依赖人工看日志判断是否完成。
- 异常场景至少覆盖模型失败、工具失败、MCP 不可用、SSE 断开中的 2 类。
- 每次执行后能输出测试报告，包含成功率、失败原因、耗时、关键 session/requestId。

## 9. 结论

当前 nuwax 智能体测试应先从链路稳定性入手，再扩展到智能体能力评测。推荐路线是：

```text
P0 主链路回归
  -> P1 智能体任务评测
  -> P2 稳定性 / 故障注入 / 安全隔离
```

这样可以先保证“智能体系统能稳定跑完”，再判断“智能体做得好不好”，最后验证“并发、异常和安全边界是否可靠”。
