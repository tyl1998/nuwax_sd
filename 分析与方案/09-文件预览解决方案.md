# skill-details 文件预览能力解决方案

## 1. 问题回顾

### 1.1 根因

| 层面 | 问题 |
|------|------|
| **数据模型** | `SkillFileInfo.contents` 是 `string` 类型，存储文本内容；二进制文件无法用纯文本表达 |
| **前端渲染** | Office 文档渲染库（`@js-preview/docx` / `@js-preview/excel` / `@js-preview/pdf` / `pptx-preview`）均需要 **URL** 或 **ArrayBuffer**，不接受纯文本 |
| **后端 API** | `/api/skill/{skillId}` 返回的 `files[]` 中可能不包含 `fileProxyUrl`（指向二进制文件的下载地址） |

### 1.2 影响范围

| 文件类型 | 当前状态 | 影响程度 |
|---------|---------|---------|
| docx / xlsx / pptx / pdf | ⚠️ 依赖 `fileProxyUrl`，缺失则无法预览 | 高 |
| doc / xls / ppt（旧格式） | ❌ 完全不支持 | 中（旧格式逐渐淘汰） |
| md / html / 代码 / txt | ✅ 正常（基于 `contents` 文本） | 无 |

---

## 2. 可行解决方案

### 方案 A：后端补全 `fileProxyUrl`（推荐 ✅）

**原理**：确保后端 API 在返回技能文件列表时，为二进制文件生成并返回 `fileProxyUrl`。

**改动范围**：

```
后端: /api/skill/{skillId} 接口
  → 对每个 SkillFileInfo:
    → 如果是二进制文件类型（docx/xlsx/pptx/pdf 等）
    → 生成 fileProxyUrl（指向文件下载地址）
    → 返回给前端
```

**前端改动**：几乎无需改动，现有的 `FilePreview` 组件已完全支持 `fileProxyUrl`。

**具体实现思路**：

1. **后端 Java/Kotlin 侧**（`SkillFileInfo` 实体）：
   ```java
   // 在文件上传/存储时，为二进制文件生成下载 URL
   // 方式一：使用 OSS 对象存储的签名 URL
   String fileProxyUrl = ossClient.generatePresignedUrl(bucket, objectKey, expiration);
   
   // 方式二：使用内部文件服务器 URL
   String fileProxyUrl = "/api/computer/static/" + computerId + "/" + filePath;
   
   // 方式三：使用 nuwax-file-server 提供的 URL
   String fileProxyUrl = fileServerBaseUrl + "/download/" + fileId;
   ```

2. **前端验证**：检查 `src/components/FileTreeView/index.tsx:629` 附近的 `fileProxyUrl` 为空的注释，确认后端确实未返回。

**优点**：
- ✅ 前端改动最小，现有渲染链路完全复用
- ✅ 所有第三方预览库均支持 URL 输入
- ✅ 方案成熟稳定

**缺点**：
- ❌ 需要后端配合改动
- ❌ 依赖文件可被 HTTP 访问（需要文件服务器或 OSS 支持）
- ❌ 需考虑 URL 有效期和鉴权

### 方案 B：前端将 Base64 `contents` 转换为 ArrayBuffer（应急方案）

**原理**：如果后端将二进制文件以 Base64 编码存储在 `contents` 字段中，前端可以解码为 ArrayBuffer 传给预览库。

**改动范围**：

1. **修改 `fetchContentFromUrl()`**（`src/services/skill.ts`）：
   ```typescript
   // 识别二进制文件类型
   const BINARY_EXTENSIONS = ['docx', 'xlsx', 'xls', 'pptx', 'ppt', 'pdf'];
   
   async function fetchSkillFileContent(file: SkillFileInfo): Promise<string | ArrayBuffer> {
     const ext = getFileExtension(file.name);
     
     if (BINARY_EXTENSIONS.includes(ext)) {
       // 方式一：后端 contents 是 Base64 编码
       const base64Content = await fetch(file.contentUrl).then(r => r.text());
       return base64ToArrayBuffer(base64Content);
       
       // 方式二：后端提供二进制下载接口
       return await fetch(file.downloadUrl).then(r => r.arrayBuffer());
     }
     
     return await fetch(file.contentUrl).then(r => r.text());
   }
   
   function base64ToArrayBuffer(base64: string): ArrayBuffer {
     const binaryString = atob(base64);
     const bytes = new Uint8Array(binaryString.length);
     for (let i = 0; i < binaryString.length; i++) {
       bytes[i] = binaryString.charCodeAt(i);
     }
     return bytes.buffer;
   }
   ```

2. **修改 `FilePreview` 组件**（`src/components/business-component/FilePreview/index.tsx`）：
   ```typescript
   // 在 Office 文档渲染分支中，增加 ArrayBuffer 支持
   // docx（@js-preview/docx）
   jsPreviewDocx.init(container).preview(arrayBuffer);  // 已支持 ArrayBuffer ✅
   
   // xlsx（@js-preview/excel）
   jsPreviewExcel.init(container).preview(arrayBuffer);  // 已支持 ArrayBuffer ✅
   
   // pdf（@js-preview/pdf）
   jsPreviewPdf.init(container).preview(arrayBuffer);    // 已支持 ArrayBuffer ✅
   
   // pptx（pptx-preview）— 需要特殊处理
   // pptx-preview 需要 URL，改为直接传 ArrayBuffer
   ```

3. **修改 `FileTreeView` 渲染逻辑**：
   ```typescript
   // 在 renderContent() 的 Office 文档分支中
   if (isOfficeDocument && (fileProxyUrl || binaryContent)) {
     return <FilePreview src={fileProxyUrl} binaryContent={binaryContent} ... />;
   }
   ```

**优点**：
- ✅ 不依赖后端改动，前端可独立完成
- ✅ 利用现有预览库的 ArrayBuffer 接口

**缺点**：
- ❌ 需要后端将二进制文件以 Base64 存储在 `contents` 中（可能增加存储和传输成本）
- ❌ Base64 编码增加约 33% 的数据量
- ❌ 大文件（>10MB）传输和解码可能影响性能
- ❌ `pptx-preview` 库不直接支持 ArrayBuffer，需要额外处理

### 方案 C：前端构建专用二进制下载接口（中等改动）

**原理**：为技能文件新增一个二进制下载接口，前端根据文件 ID 直接请求二进制数据。

**改动范围**：

1. **后端新增接口**：
   ```
   GET /api/skill/{skillId}/file/{fileId}/binary
   Response: application/octet-stream (二进制流)
   ```

2. **前端新增下载方法**：
   ```typescript
   // src/services/skill.ts
   async function fetchSkillFileBinary(skillId: string, fileId: string): Promise<ArrayBuffer> {
     const response = await fetch(`/api/skill/${skillId}/file/${fileId}/binary`, {
       cache: 'no-store',
     });
     return response.arrayBuffer();
   }
   ```

3. **修改 `FileTreeView` 和 `FilePreview`**：
   - 对二进制文件类型，先调用 `fetchSkillFileBinary()` 获取 ArrayBuffer
   - 传入 `FilePreview` 的 `binaryContent` 属性
   - `FilePreview` 内部将 ArrayBuffer 传给对应的预览库

**优点**：
- ✅ 清晰的接口设计，职责分明
- ✅ 不依赖现有 `contents` 字段，可以逐步迁移
- ✅ 支持大文件流式传输

**缺点**：
- ❌ 需要后端新增接口
- ❌ 需要前端增加新的数据获取逻辑

---

### 方案 D：文件服务器直链（利用现有基础设施）

**原理**：如果技能文件存储在 `nuwax-file-server` 或 Computer 工作空间中，可以直接构造文件服务器的 URL 作为 `fileProxyUrl`。

**改动范围**：

1. **分析文件存储位置**：
   - 技能文件是否存储在 Computer 的工作空间中？
   - 是否可以通过 `nuwax-file-server` 的 `/api/computer/static/` 路径访问？

2. **前端构造 URL**：
   ```typescript
   // 在 SkillFileInfo 转换时
   function buildFileProxyUrl(file: SkillFileInfo): string | undefined {
     if (file.filePath && file.computerId) {
       // 使用 nuwax-file-server 的静态文件服务
       return `/api/computer/static/${file.computerId}/workspace/${file.filePath}`;
     }
     return undefined;
   }
   ```

3. **补充到 `transformFlatListToTree()` 中**：
   ```typescript
   // src/components/FileTreeView/index.tsx
   const fileProxyUrl = file.fileProxyUrl || buildFileProxyUrl(file);
   ```

**优点**：
- ✅ 利用现有文件服务器基础设施
- ✅ 前端改动小

**缺点**：
- ❌ 依赖文件服务器可达性和文件路径映射关系
- ❌ 需要确认文件确实在文件服务器上有存储

---

### 方案 E：混合方案（推荐实施路径 ✅✅）

**分阶段实施，快速见效 + 长期完善**：

```
阶段 1（短期，1-2 天）：
  → 后端排查为何 fileProxyUrl 为空
  → 如果文件服务器上已有文件，补全 URL 返回逻辑
  → 前端零改动，立即生效

阶段 2（中期，3-5 天）：
  → 后端新增 /api/skill/{skillId}/file/{fileId}/binary 下载接口
  → 前端 FileTreeView 增加 ArrayBuffer 预览逻辑
  → 支持所有 Office 文档类型

阶段 3（长期优化）：
  → .doc 旧格式支持（引入 mammoth.js 或 LibreOffice 在线转换）
  → 大文件分块加载
  → 预览缓存机制
```

### 4.2 新增服务方法

```typescript
// src/services/skill.ts
const BINARY_EXTENSIONS = new Set(['docx', 'xlsx', 'xls', 'pptx', 'ppt', 'pdf']);

export async function fetchSkillFileBinary(
  skillId: string, 
  fileId: string
): Promise<ArrayBuffer> {
  const response = await fetch(`/api/skill/${skillId}/file/${fileId}/binary`, {
    method: 'GET',
    cache: 'no-store',
  });
  
  if (!response.ok) {
    throw new Error(`Failed to fetch binary file: ${response.status}`);
  }
  
  return response.arrayBuffer();
}

export function isBinaryFile(fileName: string): boolean {
  const ext = fileName.split('.').pop()?.toLowerCase() || '';
  return BINARY_EXTENSIONS.has(ext);
}
```

### 4.3 修改 FileTreeView 渲染逻辑

```typescript
// src/components/FileTreeView/index.tsx
// 在 renderContent() 中的 Office 文档分支（约 1482 行）

const [binaryContent, setBinaryContent] = useState<ArrayBuffer | null>(null);
const [binaryLoading, setBinaryLoading] = useState(false);

// 当选中二进制文件时，异步加载
useEffect(() => {
  if (selectedFileNode && isBinaryFile(selectedFileNode.name) && !selectedFileNode.fileProxyUrl) {
    setBinaryLoading(true);
    fetchSkillFileBinary(skillId, selectedFileNode.id)
      .then(buffer => setBinaryContent(buffer))
      .catch(err => console.error('Failed to load binary file:', err))
      .finally(() => setBinaryLoading(false));
  } else {
    setBinaryContent(null);
  }
}, [selectedFileNode?.id]);

// renderContent() 中修改 Office 文档分支
if (isOfficeDocument) {
  if (fileProxyUrl) {
    return <FilePreview src={fileProxyUrl} fileType={officeFileType} />;
  }
  if (binaryContent) {
    return <FilePreview binaryContent={binaryContent} fileType={officeFileType} />;
  }
  if (binaryLoading) {
    return <Spin tip="加载文档中..." />;
  }
  return <div>无法加载文档预览</div>;
}
```

## 5. 关于 `.doc` 旧格式的支持

`.doc` 是 Microsoft Word 97-2003 的二进制格式，`@js-preview/docx` 不支持。可选方案：

| 方案 | 说明 | 复杂度 |
|------|------|-------|
| **mammoth.js** | 将 `.doc` 转换为 HTML 渲染（仅支持基本格式） | 低 |
| **LibreOffice 后端转换** | 后端使用 LibreOffice 将 `.doc` → `.docx` 或 `.pdf` | 中 |
| **提示用户转换** | 前端显示"请将 .doc 文件转换为 .docx 格式后重新上传" | 无 |
| **下载后本地打开** | 提供下载按钮，用户自行下载打开 | 无 |

推荐在短期使用"提示用户转换 + 下载按钮"，长期使用"LibreOffice 后端转换"。

## 6. 最终推荐

### 立即可做（0 代码改动）

1. **排查后端**：确认 `/api/skill/{skillId}` 返回的 `files[]` 中，二进制文件的 `fileProxyUrl` 字段是否为空
2. **如果为空**：定位原因 — 是后端未生成 URL，还是前端字段名映射错误
3. **如果后端已有文件存储**：补全 URL 生成逻辑，前端零改动即可生效

### 短期方案（1-2 周）

**方案 C（新增二进制下载接口）+ 方案 D（文件服务器直链）混合使用**：
- 优先使用 `fileProxyUrl`（方案 D）
- 降级使用 `/api/skill/{skillId}/file/{fileId}/binary`（方案 C）
- `.doc` 格式提供下载按钮 + 提示信息

### 长期优化

- 大文件预览性能优化（虚拟滚动、懒加载）
- 预览结果缓存（避免重复下载）
- LibreOffice 在线转换服务（支持所有 Office 格式）
- 文件预览组件独立为通用微前端模块
