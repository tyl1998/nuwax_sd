## 2. 工程是怎么跑起来的

程序入口在 `src/main.rs` 的 `main()` 函数，启动流程如下：

```
main()
  ├── 1. Config::load_config()          // 加载配置
  ├── 2. init_tracing()                 // 初始化日志（控制台 + 文件）
  ├── 3. AppStates::new(config)         // 创建全局状态（reqwest Client）
  ├── 4. tokio::spawn(create_indexes)   // 异步创建 Quickwit 索引（带重试）
  │     ├── ensure_agent_index_exists()
  │     ├── ensure_knowledge_index_exists()
  │     └── ensure_common_log_index_exists()
  ├── 5. tokio::spawn(migrate_data)     // 异步执行 v1→v2 数据迁移
  ├── 6. init_routes(app_states)        // 注册所有路由
  └── 7. axum::serve(listener, app)     // 启动 HTTP 服务
```

关键设计：**索引创建和数据迁移都是异步 spawn 的**，不会阻塞服务启动。即使 Quickwit 暂时不可用，服务也能正常启动，后续通过重试机制自动恢复。

## 3. 配置加载机制

配置结构体：

```rust
pub struct Config {
    pub server: ServerConfig,
    pub quickwit: QuickwitConfig,
}

pub struct ServerConfig {
    pub port: u16,        // 默认 8098
    pub log_path: String, // 默认 "./logs"
}

pub struct QuickwitConfig {
    pub url: String,      // 默认 "http://127.0.0.1:7280"
}
```

### 配置优先级（从高到低）

| 优先级 | 来源 | 说明 |
|--------|------|------|
| 1 | 环境变量 `LOG_PLATFORM_CONFIG` | 指定自定义配置文件路径 |
| 2 | `/app/config.yml` | Docker 容器内的配置文件 |
| 3 | `./config.yml` | 本地开发配置文件 |
| 4 | 环境变量覆盖 | `LOG_PLATFORM_PORT`、`LOG_PLATFORM_LOG_PATH`、`QUICKWIT_URL` |

环境变量覆盖发生在配置文件加载之后，即使配置文件中已有值，环境变量也能覆盖。

### 本地开发配置（config.yml）

```yaml
server:
  port: 8098
  log_path: "./logs"
quickwit:
  url: "http://127.0.0.1:7280"
```

### Docker 生产配置（docker/config.docker.yml）

```yaml
server:
  port: 8097
  log_path: "/app/logs"
quickwit:
  url: "http://host.docker.internal:7280"
```

## 4. 全局状态（AppStates）

`AppStates` 是整个服务的共享状态，通过 `Arc` 在所有 handler 中共享：

```rust
pub struct AppStates {
    pub http_client: reqwest::Client,     // 30s 总超时，10s 连接超时
    pub quickwit_config: Arc<QuickwitConfig>,
}
```

`http_client` 的超时配置：
- **总超时**：30 秒（防止长时间挂起的请求）
- **连接超时**：10 秒（快速失败）

## 5. 中间件栈

请求处理链路（从外到内）：

```
客户端请求
  │
  ├── 1. ServerTimeLayer     → 在响应头注入 x-server-time
  ├── 2. TraceLayer          → tower-http 的请求追踪
  ├── 3. request_logger      → 记录请求方法、路径、耗时
  ├── 4. request_id          → 注入/透传 x-request-id
  │
  ▼
路由匹配 → Handler 处理
```

### CORS 配置

- 允许任意 Origin（`any()`）
- 允许任意 Method
- 允许任意 Header
- 最大请求体：**20 MB**（适配批量日志写入场景）

## 6. 索引创建与重试机制

服务启动时会异步创建 3 个 Quickwit 索引：

| 索引 | 默认名称 | 说明 |
|------|----------|------|
| Agent 日志 | `agent_logs` | v1 索引，后续可迁移至 `agent_logs_v2` |
| 知识库片段 | `knowledge_segments` | 知识库向量片段 |
| 通用日志 | `common_logs` | 通用运行日志 |

重试策略：
- 最多重试 **5 次**
- 每次间隔 **3 秒**
- 先检查索引是否已存在（`check_index_exists`），存在则跳过
- 所有重试失败后记录错误日志，**不阻塞服务启动**

## 7. 数据迁移（v1 → v2）

Agent 日志支持索引版本迁移：

1. 使用 `redb` 嵌入式数据库持久化迁移检查点（checkpoint）
2. 从旧索引批量读取数据
3. 写入新索引（`agent_logs_v2`）
4. 更新 checkpoint，支持断点续传
5. 通过 API 查询迁移状态、重置迁移

迁移相关 API：
- `POST /api/agent/log/migrate` — 触发迁移
- `GET /api/agent/log/migration/status` — 查询迁移状态
- `POST /api/agent/log/migration/reset` — 重置迁移

## 8. 从运行时角度如何理解这个服务

### 配置层
YAML 文件 + 环境变量覆盖，配置驱动，不硬编码任何地址。

### 状态层
单一 `AppStates` 持有 HTTP 客户端和配置引用，通过 `Arc` 共享，无全局可变状态。

### 路由层
Axum Router 集中定义，按业务域分组（agent / knowledge / common），每个 handler 通过 `State` 提取器获取共享状态。

### 服务层
三个 Service 结构体分别封装与 Quickwit 的交互逻辑，负责索引管理、数据写入、搜索查询。所有 Quickwit 交互都是 HTTP 调用（reqwest），无 SDK 依赖。

### 异步任务层
索引创建和数据迁移通过 `tokio::spawn` 在后台执行，与主服务解耦，失败可重试。
