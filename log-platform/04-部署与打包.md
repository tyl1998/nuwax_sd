# log-platform 部署与打包

## 1. 这是一个标准 Rust 服务容器化部署模型

log-platform 的部署模型很简单：

- **多阶段 Docker 构建**：Rust 编译阶段 → Debian 运行时阶段
- **配置外置**：通过 YAML 文件 + 环境变量注入，不硬编码任何地址
- **单容器运行**：一个容器跑 log-platform，Quickwit 作为外部服务独立部署

## 2. Dockerfile 分析

位于 `docker/Dockerfile`，采用两阶段构建：

### 阶段一：Builder（编译）

```dockerfile
FROM rust:1.90-slim AS builder
WORKDIR /app
# 1. 复制 Cargo.toml 和 Cargo.lock，先下载依赖（利用 Docker 缓存层）
COPY Cargo.toml Cargo.lock ./
# 2. 创建空 src/lib.rs 触发依赖下载
RUN mkdir src && echo "fn main() {}" > src/main.rs && cargo build --release && rm -rf src
# 3. 复制完整源码，重新编译
COPY src ./src
RUN cargo build --release
```

关键设计：**依赖缓存层**。先只复制 `Cargo.toml` 和 `Cargo.lock`，创建空 `main.rs` 触发依赖下载。之后复制完整源码重新编译时，如果依赖没变，`cargo build` 只重新编译业务代码，大幅加速构建。

### 阶段二：Runtime（运行时）

```dockerfile
FROM debian:bookworm-slim
RUN apt-get update && apt-get install -y --no-install-recommends ca-certificates && rm -rf /var/lib/apt/lists/*
WORKDIR /app
COPY --from=builder /app/target/release/log-platform .
COPY docker/config.docker.yml ./config.yml
EXPOSE 8097
HEALTHCHECK --interval=30s --timeout=3s CMD curl -f http://localhost:8097/health || exit 1
CMD ["./log-platform"]
```

关键点：
- 使用 `debian:bookworm-slim` 而非 `alpine`，因为 Rust 编译产物依赖 glibc
- 安装 `ca-certificates` 以支持 HTTPS 请求（调用外部 Quickwit）
- 健康检查每 30 秒执行一次，超时 3 秒
- 暴露端口 **8097**（生产环境）

## 3. 配置文件优先级

| 优先级 | 来源 | 场景 |
|--------|------|------|
| 最高 | 环境变量 `LOG_PLATFORM_CONFIG` | 自定义配置文件路径 |
| 中 | `/app/config.yml` | Docker 容器内（COPY 进去的） |
| 低 | `./config.yml` | 本地开发 |
| 覆盖 | 环境变量 `LOG_PLATFORM_PORT` | 覆盖配置文件中的端口 |
| 覆盖 | 环境变量 `LOG_PLATFORM_LOG_PATH` | 覆盖日志路径 |
| 覆盖 | 环境变量 `QUICKWIT_URL` | 覆盖 Quickwit 地址 |

### 生产环境配置（docker/config.docker.yml）

```yaml
server:
  port: 8097
  log_path: "/app/logs"
quickwit:
  url: "http://host.docker.internal:7280"
```

### 本地开发配置（config.yml）

```yaml
server:
  port: 8098
  log_path: "./logs"
quickwit:
  url: "http://127.0.0.1:7280"
```

## 4. Docker 构建命令

```bash
# 在 log-platform/ 根目录执行
docker build -f docker/Dockerfile -t log-platform:latest .

# 带版本标签
docker build -f docker/Dockerfile -t log-platform:$(git rev-parse --short HEAD) .
```

## 5. 容器启动命令

### 生产环境（Jenkins 单容器启动）

```bash
docker run -d \
  --name log-platform \
  --restart unless-stopped \
  -p 8097:8097 \
  -e QUICKWIT_URL="http://host.docker.internal:7280" \
  -v /data/log-platform/logs:/app/logs \
  log-platform:latest
```

参数说明：

| 参数 | 说明 |
|------|------|
| `--restart unless-stopped` | 异常退出自动重启 |
| `-p 8097:8097` | 映射生产端口 |
| `-e QUICKWIT_URL` | 通过环境变量注入 Quickwit 地址 |
| `-v /data/log-platform/logs:/app/logs` | 日志目录持久化到宿主机 |

### 本地开发

```bash
# 直接 cargo run（使用 config.yml）
cargo run

# 或指定 Quickwit 地址
QUICKWIT_URL=http://localhost:7280 cargo run
```

## 6. 环境变量速查

| 环境变量 | 默认值 | 说明 |
|----------|--------|------|
| `LOG_PLATFORM_CONFIG` | 无 | 自定义配置文件路径 |
| `LOG_PLATFORM_PORT` | 8098（本地）/ 8097（Docker） | 服务监听端口 |
| `LOG_PLATFORM_LOG_PATH` | `./logs`（本地）/ `/app/logs`（Docker） | 日志文件存储路径 |
| `QUICKWIT_URL` | `http://127.0.0.1:7280`（本地）/ `http://host.docker.internal:7280`（Docker） | Quickwit 服务地址 |

## 7. 与其他服务的网络关系

```
┌─────────────────────────────────────────────┐
│              Docker Host                     │
│                                              │
│  ┌──────────────┐    ┌──────────────┐       │
│  │ log-platform  │    │   Quickwit   │       │
│  │   :8097       │───▶│   :7280      │       │
│  └──────────────┘    └──────────────┘       │
│         ▲                                    │
│         │ host.docker.internal               │
│  ┌──────────────┐                           │
│  │ nuwax-backend │                          │
│  │  (调用方)     │                           │
│  └──────────────┘                           │
└─────────────────────────────────────────────┘
```

- log-platform 通过 `host.docker.internal` 访问宿主机上的 Quickwit
- nuwax-backend、rcoder 等调用方通过 `host.docker.internal:8097` 访问 log-platform
- 这种模式下每个服务独立容器，通过宿主机网络互通

## 8. 健康检查

| 端点 | 方法 | 说明 |
|------|------|------|
| `/health` | GET | 就绪检查，返回 200 表示服务可用 |
| `/ready` | GET | 就绪检查（别名） |

Docker HEALTHCHECK 配置：
```dockerfile
HEALTHCHECK --interval=30s --timeout=3s CMD curl -f http://localhost:8097/health || exit 1
```

注意：服务启动后即使 Quickwit 暂时不可用，`/health` 也会返回 200。索引创建失败不会阻塞服务启动，后台会自动重试。

## 9. 日志输出

服务启动时通过 `tracing-subscriber` 初始化日志：
- **控制台输出**：`fmt::layer()` 格式化输出到 stdout
- **文件输出**：写入配置的 `log_path` 目录
- **日志级别**：通过 `RUST_LOG` 环境变量控制（默认 `info`）

```bash
# 启用 debug 日志
RUST_LOG=debug cargo run

# 只看 log-platform 的 debug 日志
RUST_LOG=log_platform=debug cargo run
```

## 10. 从部署角度怎么理解这个服务

### 它是一个无状态服务
log-platform 本身不存储业务数据，所有数据都在 Quickwit 中。本地 `redb` 只存迁移检查点，丢失不影响业务。

### 它是一个中间层代理
对上层（nuwax-backend 等）提供 HTTP API，对下层（Quickwit）发起 HTTP 请求，本身不做数据持久化。

### 配置完全外置
不硬编码任何地址，通过 YAML 文件 + 环境变量灵活配置，适配本地开发、Docker、K8s 等各种部署场景。

### 构建产物很小
多阶段构建只复制编译好的二进制文件和配置文件，最终镜像基于 `debian:bookworm-slim`，体积小、启动快。
