# nuwax-cli 分布式与内部平台改造关注点

这篇不讲“现在怎么工作”，而是直接讲：

如果后续你们要把 `nuwax-cli` 改造成适配公司内部平台、支持分布式部署，当前实现里有哪些天然耦合点。

## 1. 当前实现默认的是“单机部署模型”

从 [run_auto_upgrade_deploy](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs) 看，当前部署模型默认成立的前提是：

1. 命令执行机就是部署机
2. 命令执行机本地有 `docker/` 工作目录
3. 命令执行机本地能直接访问 Docker / Compose
4. 升级时可以在本机先停服务，再覆盖目录，再重启

这本质上是：

- “单机原地升级”
- 不是“控制面与执行面分离”
- 更不是“多节点编排”

## 2. 需要优先拆开的耦合

## 2.1 本地文件系统耦合

体现位置：

- [init.rs:25-33](../../nuwax-cli/nuwax-cli/src/init.rs)
- [auto_upgrade_deploy.rs:266-336](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)
- [client-core/src/container/volumes.rs:19-45](../../nuwax-cli/client-core/src/container/volumes.rs)

当前假设：

- 制品下载到本地目录
- 部署包解压到本地 `docker/`
- bind mount 路径在本地直接创建

对内部平台的影响：

- 如果公司平台采用制品仓 + 远端执行器模型，这些本地路径假设都要抽象掉

建议改造方向：

- 把“制品下载”和“制品落地”抽成独立接口
- 支持“本地文件系统实现”和“平台制品仓实现”

## 2.2 本地 Docker / Compose 耦合

体现位置：

- [client-core/src/container/command.rs:12-140](../../nuwax-cli/client-core/src/container/command.rs)
- [client-core/src/container/environment.rs:29-74](../../nuwax-cli/client-core/src/container/environment.rs)

当前假设：

- 当前机器装了 `docker` 或 `docker-compose`
- 当前进程有权限执行它们
- 当前进程能直接感知容器运行状态

对内部平台的影响：

- 如果平台是“调用内部发布 API”而不是直接 shell Docker，这层必须重做

建议改造方向：

- 把 `DockerManager` 抽象成更上层的“部署执行器”
- 至少支持两种实现：
  - Compose 本地执行器
  - 公司平台远端执行器

## 2.3 本机停服再替换的升级语义

体现位置：

- [auto_upgrade_deploy.rs:216-231](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)
- [docker_service.rs:269-377](../../nuwax-cli/nuwax-cli/src/commands/docker_service.rs)

当前假设：

- 升级时可以先停本机服务
- 然后再替换文件
- 再重新启动

这对单机没问题，但对分布式平台通常不够：

- 你们可能需要滚动升级
- 需要分批 drain 节点
- 需要健康门禁
- 需要回滚到上一批而不是整机回滚

建议改造方向：

- 把“stop -> replace -> start”改造成可插拔策略
- 当前策略保留为 `single_host_inplace`
- 新增 `rolling` / `platform_managed` 策略

## 2.4 配置状态写在本地 `config.toml`

体现位置：

- [client-core/src/config.rs](../../nuwax-cli/client-core/src/config.rs)
- [auto_upgrade_deploy.rs:60-77](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)

当前假设：

- 当前 Docker 服务版本由本地配置文件记录
- 部署完成后直接修改本地版本

对分布式平台的影响：

- 多节点下，本地配置不能天然代表全局真实部署状态
- 平台更可能要求状态写到控制平面，而不是某台机器的 TOML

建议改造方向：

- 区分“本地期望版本”和“平台真实部署版本”
- 版本状态存储抽象化：
  - 本地文件实现
  - 平台状态中心实现

## 2.5 健康检查直接面向本机容器

体现位置：

- [status.rs:139-165](../../nuwax-cli/nuwax-cli/src/commands/status.rs)
- [docker_service/health_check.rs](../../nuwax-cli/nuwax-cli/src/docker_service/health_check.rs)

当前假设：

- 健康状态 = 当前机器上的容器状态

对分布式平台的影响：

- 平台层健康状态往往来自：
  - 平台调度状态
  - 服务发现
  - 业务探针
  - 多节点汇总

建议改造方向：

- 把 `health_check` 分成本机容器健康和平台服务健康两层

## 4. 我建议的改造拆分方式

如果后续真的要做兼容公司内部平台，建议不要一把梭重写，而是先按边界拆。

## 第一层：保留 CLI，抽掉执行后端

目标：

- CLI 仍然保留
- 但把底层“如何部署”从 Compose 命令改成接口

建议抽象：

- `ArtifactProvider`
- `DeploymentExecutor`
- `DeploymentStateStore`
- `HealthProbe`

## 第二层：把单机部署流程变成可选策略

当前策略：

- `single_host_inplace`

未来策略：

- `multi_host_rolling`
- `internal_platform_release`

## 第三层：把本地状态和平台状态分开

本地状态：

- 缓存
- 临时文件
- 本地调试信息

平台状态：

- 发布版本
- 节点批次
- 发布结果
- 健康状态

## 5. 当前最值得重点盯住的改造入口

如果后面要落代码，我建议优先看这 4 个位置：

1. [run_auto_upgrade_deploy](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)
2. [deploy_docker_services](../../nuwax-cli/nuwax-cli/src/commands/docker_service.rs)
3. [DockerManager::run_compose_command](../../nuwax-cli/client-core/src/container/command.rs)
4. [DockerManager::with_project](../../nuwax-cli/client-core/src/container/config.rs)

这 4 个位置几乎就把：

- 调度入口
- 部署执行
- Compose 调用
- 部署目标描述

全串起来了。

## 6. 最短结论

`nuwax-cli` 当前是一个“本地单机 Docker Compose 部署器”，带升级、备份、权限修复和健康检查增强。

如果你们要改造成：

- 分布式部署
- 公司内部平台托管部署

那最重要的不是改命令行，而是把这些强耦合拆出去：

1. 本地文件系统
2. 本地 Docker/Compose 执行
3. 单机停服再升级
4. 本地配置即部署真相
5. 本机容器健康即服务真相

这 5 个点拆开之后，CLI 才能真正变成“控制面”，而不只是“单机执行器”。
