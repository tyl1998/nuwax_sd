# nuwax-cli 关键流程梳理

这篇把常见命令和内部调用链连起来。

## 1. 初始化流程 `init`

命令入口：

- [main.rs](../../nuwax-cli/nuwax-cli/src/main.rs)
- [init.rs](../../nuwax-cli/nuwax-cli/src/init.rs)

流程：

```text
nuwax-cli init
  -> main.rs 特判 Commands::Init
  -> run_init(force)
  -> AppConfig::default
  -> save_to_file("config.toml")
  -> 创建目录
  -> Database::connect
  -> database.init_database()
  -> ApiClient::register_client()
  -> 保存 server client_id
```

外部副作用：

- 创建 `config.toml`
- 创建本地目录
- 创建 DuckDB
- 向远端服务注册客户端

为什么它独立处理：

- 初始化前还没有稳定配置和数据库，不能按普通命令走 `CliApp`

## 2. 状态检查流程 `status`

命令入口：

- [main.rs](../../nuwax-cli/nuwax-cli/src/main.rs)
- [commands/status.rs](../../nuwax-cli/nuwax-cli/src/commands/status.rs)

流程：

```text
nuwax-cli status
  -> main.rs 特判 Commands::Status
  -> show_client_version()
  -> 尝试 CliApp::new_with_auto_config()
  -> run_status_details(app)
  -> 检查 config / compose / 服务包文件
  -> HealthChecker::health_check()
  -> 输出建议
```

外部副作用：

- 基本无写操作
- 主要读配置、读数据库、查 Docker 状态

为什么要这样设计：

- 状态命令常用于排障，哪怕初始化失败也应该尽量有输出

## 3. 服务包下载与升级策略流程 `upgrade`

命令入口：

- [commands/update.rs](../../nuwax-cli/nuwax-cli/src/commands/update.rs)
- [client-core/src/upgrade.rs](../../nuwax-cli/client-core/src/upgrade.rs)

流程：

```text
nuwax-cli upgrade
  -> run_upgrade(app, args)
  -> UpgradeManager::check_for_updates(force)
  -> ApiClient::get_enhanced_service_manifest()
  -> UpgradeStrategyManager::determine_strategy()
  -> FullUpgrade / PatchUpgrade / NoUpgrade
  -> 下载到版本缓存目录
  -> 根据魔数判断格式
  -> 生成最终归档文件名
```

关键点：

- 不是所有升级都立刻部署
- 它更像“升级包准备器 + 策略判断器”

相关模块：

- [client-core/src/upgrade_strategy.rs](../../nuwax-cli/client-core/src/upgrade_strategy.rs)
- [client-core/src/version.rs](../../nuwax-cli/client-core/src/version.rs)

## 4. 离线下载流程 `upgrade download`

命令入口：

- [main.rs](../../nuwax-cli/nuwax-cli/src/main.rs)
- [commands/update.rs](../../nuwax-cli/nuwax-cli/src/commands/update.rs)

流程：

```text
nuwax-cli upgrade download
  -> main.rs 特判
  -> run_download(config_path)
  -> 尝试加载配置，失败则用默认配置
  -> ApiClient::get_enhanced_service_manifest()
  -> 按本机架构下载全量包
```

为什么单独特判：

- 离线下载不依赖 DuckDB，不该因为数据库初始化失败而不可用

## 5. Docker 服务控制流程 `docker-service`

命令入口：

- [commands/docker_service.rs](../../nuwax-cli/nuwax-cli/src/commands/docker_service.rs)

流程：

```text
nuwax-cli docker-service start|stop|restart|status
  -> run_docker_service_command()
  -> 组装 DockerService
  -> 必要时自定义 compose_path / project_name
  -> 调用 start_services / stop_services / health_check
```

关键点：

- CLI 层还会处理 `project_name` 和自定义 compose 文件路径
- 真正底层执行依赖 `DockerManager`

相关模块：

- [client-core/src/container/](../../nuwax-cli/client-core/src/container/mod.rs)
- [nuwax-cli/src/docker_service/](../../nuwax-cli/nuwax-cli/src/docker_service/mod.rs)

## 6. 备份流程 `backup`

命令入口：

- [commands/backup.rs](../../nuwax-cli/nuwax-cli/src/commands/backup.rs)
- [client-core/src/backup.rs](../../nuwax-cli/client-core/src/backup.rs)

流程：

```text
nuwax-cli backup
  -> run_backup(app)
  -> DockerService::health_check()
  -> 判断持续型服务是否已停止
  -> 选择 source_paths
  -> BackupManager::create_backup()
  -> 写备份文件
  -> 写备份记录到 DuckDB
```

这里很值得注意的一点：

- 备份不是盲备
- 它会先区分“持续型服务”和“一次性任务容器”
- 只有前者仍在运行时，才阻止冷备份

## 7. 自动升级部署流程 `auto-upgrade-deploy run`

命令入口：

- [main.rs](../../nuwax-cli/nuwax-cli/src/main.rs)
- [commands/auto_upgrade_deploy.rs](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)

流程：

```text
nuwax-cli auto-upgrade-deploy run
  -> main.rs 先做 CLI 自更新检查
  -> run_auto_upgrade_deploy()
  -> 获取最新版本
  -> run_upgrade() 下载服务包
  -> 判断首次部署 / 升级部署
  -> 升级部署时先停服务
  -> 检查并创建挂载目录
  -> 解压/替换
  -> 更新配置中的版本
  -> 启动服务并验证
```

这个流程是整个仓库里最“运维编排化”的一条主流程。

它串起了：

- 版本检查
- 下载
- 停服务
- 文件替换
- 数据库 / 配置更新
- 启服务
- 校验

## 8. CLI 自更新流程 `check-update`

命令入口：

- [commands/check_update.rs](../../nuwax-cli/nuwax-cli/src/commands/check_update.rs)

流程：

```text
nuwax-cli check-update check|install
  -> UpdateSourceManager
  -> 优先请求版本检查服务器
  -> 失败则回退 GitHub API
  -> 解析 release / updater 响应
  -> 选择当前平台对应资产
  -> 下载并自替换
```

关键点：

- CLI 本身也有自己的更新逻辑
- 和 Docker 服务升级逻辑是两条线

## 9. 你读流程时可以重点观察的 3 个设计选择

### 特判命令很多

说明作者不想让“数据库没好、配置没好”这种问题把所有命令都拖死。

### `CliApp` 很像依赖注入容器

说明这个项目虽然是 CLI，但已经是一个“小型应用”，不是几个孤立命令的拼接。

### 升级流程比普通 CLI 更重

说明 `nuwax-cli` 的核心价值不只是“管理命令”，而是“把升级部署这件高风险操作流程化”。
