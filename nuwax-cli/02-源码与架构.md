# nuwax-cli 源码与架构

这篇面向研发，目标是快速建立“代码地图”。

## 1. Workspace 结构

顶层 workspace 定义在：

- [../nuwax-cli/Cargo.toml](../../nuwax-cli/Cargo.toml)

当前只有两个 member：

- `client-core`
- `nuwax-cli`

这是一个很典型的“两层结构”：

- `nuwax-cli` 负责 CLI、交互、命令路由
- `client-core` 负责复用性更高的业务逻辑和底层能力

## 2. CLI 层怎么组织

### 入口文件

- [nuwax-cli/src/main.rs](../../nuwax-cli/nuwax-cli/src/main.rs)

这是真正的程序入口。

它做的几件事很关键：

1. 初始化 rustls
2. 固定 CLI 语言
3. 解析参数
4. 对部分命令做特殊分流
5. 初始化 `CliApp`
6. 调用 `app.run_command(...)`

### 命令定义

- [nuwax-cli/src/cli.rs](../../nuwax-cli/nuwax-cli/src/cli.rs)

这里定义了所有用户可见命令：

- `status`
- `init`
- `check-update`
- `upgrade`
- `list-backups`
- `rollback`
- `docker-service`
- `ducker`
- `auto-backup`
- `auto-upgrade-deploy`
- `cache`
- `diff-sql`

如果你想判断“某个功能是顶层命令、子命令，还是兼容旧参数”，这里是第一入口。

### 应用装配与命令分发

- [nuwax-cli/src/app.rs](../../nuwax-cli/nuwax-cli/src/app.rs)

`CliApp` 里持有这些核心对象：

- `config`
- `database`
- `api_client`
- `authenticated_client`
- `docker_manager`
- `backup_manager`
- `upgrade_manager`

这意味着：CLI 层其实很薄，它主要做“组装依赖 + 分发命令”。

## 3. `main.rs` 里有哪些特殊分支

这个文件值得重点看，因为它不是单纯的 `Cli::parse -> run`。

### `init` 特判

原因：

- 初始化时还没有配置和数据库，不能走完整的 `CliApp::new_with_config_path`

### `status` 特判

原因：

- 即使应用初始化失败，也要尽可能输出基本状态，帮助排障

### `diff-sql` 特判

原因：

- 它只是文件级 SQL diff，不需要数据库初始化

### `auto-upgrade-deploy` 提前做 CLI 自更新检查

原因：

- 自动升级部署是重流程命令，作者希望先保证 CLI 自己是最新的

### `upgrade download` 特判

原因：

- 这是离线包下载，不需要数据库初始化

这几个分支是理解“为什么这个 CLI 不完全走同一套路”的关键。

## 6. 典型控制流

### 典型命令流

```text
Cli::parse
  -> main.rs 预处理
  -> CliApp::new_with_config_path
  -> CliApp::run_command
  -> commands::xxx
  -> client-core::xxx
  -> Docker / API / DuckDB / 文件系统
```

### 初始化流

```text
main.rs
  -> run_init
  -> AppConfig::default + save_to_file
  -> Database::connect
  -> database.init_database
  -> ApiClient::register_client
```

### 升级流

```text
run_upgrade
  -> UpgradeManager::check_for_updates
  -> ApiClient::get_enhanced_service_manifest
  -> UpgradeStrategyManager::determine_strategy
  -> 下载服务包
```
