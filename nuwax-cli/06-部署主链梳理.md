# nuwax-cli 部署主链梳理

这篇只回答一个问题：

`nuwax-cli` 是如何把“本地有个工作目录”一步步推进到“Docker 服务真正跑起来”的？

## 1. 部署主链结论

从代码看，`nuwax-cli` 的部署主链不是 `docker-service start`，而是：

```text
init
  -> upgrade / upgrade download
  -> auto-upgrade-deploy run
     -> stop services
     -> prepare docker directory
     -> extract package
     -> update config version
     -> deploy docker services
     -> health check
```

如果只看“真正完成部署”的闭环，最核心入口其实是：

- [run_auto_upgrade_deploy](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)

而不是单独的 `start` / `stop`。

## 3. 真正的部署时序

## 3.1 初始化工作目录

入口：

- [run_init](../../nuwax-cli/nuwax-cli/src/init.rs)

输出：

- `config.toml`
- `docker/`
- 备份目录
- 缓存目录
- DuckDB

这一步不是“部署服务”，但它是部署的前置地基。

## 3.2 下载服务包

入口：

- [run_upgrade](../../nuwax-cli/nuwax-cli/src/commands/update.rs)

作用：

- 获取远端 manifest
- 判断全量还是增量
- 下载归档到缓存目录

这里还没真正修改宿主机当前 `docker/` 目录，只是在准备材料。

## 3.3 自动升级部署主流程

入口：

- [run_auto_upgrade_deploy](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)

主流程可以拆成 7 步。

### 第 1 步：拉取最新版本信息

关键代码：

- [auto_upgrade_deploy.rs:192-207](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)

行为：

- 通过 `app.api_client.get_enhanced_service_manifest()` 拿最新版本
- 失败则退回到本地配置中的当前版本

### 第 2 步：调用 `run_upgrade` 下载归档

关键代码：

- [auto_upgrade_deploy.rs:209-214](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)

说明：

- 部署流程复用了升级下载逻辑
- “下载”和“部署”没有彻底分层，但调用关系很直接

### 第 3 步：判断首次部署还是升级部署

关键代码：

- [auto_upgrade_deploy.rs:216-231](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)

行为：

- 首次部署：直接继续
- 升级部署：先调用 `stop_docker_services_and_wait`

这说明它默认是“单机原地升级模型”，不是蓝绿、灰度或双活模型。

### 第 4 步：检查并创建挂载目录

关键代码：

- [auto_upgrade_deploy.rs:233-260](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)
- [client-core/src/container/volumes.rs:19-45](../../nuwax-cli/client-core/src/container/volumes.rs)

行为：

- 解析 compose 里的 bind mount
- 自动创建宿主机目录
- 对 Podman Desktop / Windows 特殊兼容

这一层很关键，因为它说明 `nuwax-cli` 不是单纯调用 compose，而是在为“宿主机准备条件”兜底。

### 第 5 步：清理旧 `docker/` 并解压新包

关键代码：

- [auto_upgrade_deploy.rs:262-350](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)

行为：

- 全量升级：清理旧 `docker/`
- 增量升级：只清理变更文件
- 解压新归档
- 修复脚本权限
- 更新配置中的 Docker 服务版本

这一步说明部署包的落点是本地 `docker/` 目录，而不是某个远端制品仓直接被容器引用。

### 第 6 步：执行真正的 Docker 部署

关键代码：

- [auto_upgrade_deploy.rs:352-359](../../nuwax-cli/nuwax-cli/src/commands/auto_upgrade_deploy.rs)
- [deploy_docker_services](../../nuwax-cli/nuwax-cli/src/commands/docker_service.rs)

行为：

- 构造 `DockerServiceManager`
- 处理可选 `config_file` / `project_name`
- 调 `deploy_services()`

### 第 7 步：健康检查与状态输出

关键代码：

- [docker_service.rs:125-166](../../nuwax-cli/nuwax-cli/src/commands/docker_service.rs)
- [health_check.rs](../../nuwax-cli/nuwax-cli/src/docker_service/health_check.rs)

行为：

- 部署完成后做 health check
- 输出容器运行数、总体状态、容器详情

## 4. Docker 层真正怎么跑

### 4.1 Compose 命令选择

代码：

- [client-core/src/container/environment.rs:29-74](../../nuwax-cli/client-core/src/container/environment.rs)
- [client-core/src/container/command.rs:12-80](../../nuwax-cli/client-core/src/container/command.rs)

行为：

- 优先尝试 `docker compose version`
- 不行再退回 `docker-compose --version`

这层是兼容性增强，不影响部署语义，但影响平台接入方式。

### 4.2 Compose 文件和项目名处理

代码：

- [client-core/src/container/config.rs:27-53](../../nuwax-cli/client-core/src/container/config.rs)
- [client-core/src/container/config.rs:170-203](../../nuwax-cli/client-core/src/container/config.rs)

行为：

- `DockerManager::with_project` 保存 `compose_file`、`env_file`、`project_name`
- `project_name` 优先使用命令传入
- 否则从 compose 配置读取 `name`
- 再否则默认 `docker`

这说明它已经有一点“多实例隔离”的雏形，但还停留在 Compose 项目名层面。

### 4.3 端口和挂载处理

代码：

- [docker_service/port_manager.rs](../../nuwax-cli/nuwax-cli/src/docker_service/port_manager.rs)
- [client-core/src/container/volumes.rs](../../nuwax-cli/client-core/src/container/volumes.rs)

行为：

- 解析 `.env`
- 展开环境变量
- 分析 host port / container port 映射
- 解析 bind mount 并创建宿主机目录

这两块对未来分布式部署改造特别关键，因为它们都默认“部署节点本地有完整文件系统语义”。

## 5. 部署视角下最关键的结论

### 结论 1

`nuwax-cli` 当前是“单机本地工作目录 + 本地 Docker Compose 编排”的部署器。

### 结论 2

它的部署不是只执行 compose，而是显式包含：

- 下载制品
- 落地文件
- 停服务
- 清理旧目录
- 修复权限
- 创建挂载目录
- 起服务
- 校验健康

### 结论 3

它已经支持一定程度的多环境参数化：

- 自定义 compose 文件
- 自定义 project name
- 自定义前端端口

但这些还不足以直接支撑“分布式多节点编排”。
